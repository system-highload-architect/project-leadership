# Событийно-ориентированная архитектура (Event-Driven Architecture, EDA)

Событийно-ориентированная архитектура — это подход, при котором компоненты системы общаются друг с другом через **асинхронные события**. Вместо прямых синхронных вызовов (REST, gRPC) сервисы публикуют события о том, что произошло, и реагируют на события, которые им интересны.

EDA — это **не просто «использование брокера сообщений»**. Это целая парадигма мышления, которая требует переосмысления того, как данные и команды движутся в системе.

## 🧠 Что такое событийно-ориентированная архитектура?

EDA строится вокруг **трёх ключевых элементов**:

1. **Событие (Event)** — факт, который произошёл в системе (например, «робот завершил доставку»). Событие неизменяемо и содержит информацию о том, что случилось.
2. **Брокер событий (Event Broker)** — промежуточное звено (Kafka, RabbitMQ, NATS), которое принимает события и доставляет их подписчикам.
3. **Подписчики (Subscribers)** — компоненты, которые реагируют на определённые события и выполняют свои действия.

## 🧩 Основные паттерны EDA

### 1. Простая EDA (basic-eda)

Компоненты общаются через брокер сообщений. Один публикует событие, другие подписываются и реагируют.

🔹 **Когда использовать:** Когда нужно развязать компоненты, избежать синхронных цепочек вызовов.

🔸 **Преимущество:** Слабая связанность, асинхронность, устойчивость.

---

### 2. Event Sourcing (event-sourcing/)

Вместо хранения текущего состояния сущности — хранение последовательности всех событий, которые с ней произошли.

🔹 **Когда использовать:** Когда важна история изменений, аудит, возможность восстановления состояния на любой момент времени.

🔸 **Особенности:** Мощный, но сложный паттерн. Требует переосмысления способа работы с данными.

---

### 3. CQRS + Event Sourcing (cqrs-event-sourcing/)

Разделение моделей для чтения и записи + хранение событий.

🔹 **Когда использовать:** Для систем с высокой нагрузкой, где чтение и запись сильно отличаются по объёму и структуре.

🔸 **Преимущество:** Максимальная гибкость и производительность для сложных доменов.

---

### 4. Actor Model (actor-model/)

Каждая единица логики — актор. Акторы обмениваются сообщениями и обрабатывают их по очереди, без общего состояния.

🔹 **Когда использовать:** Для систем реального времени, с высокой конкурентностью (например, игры, телеком).

🔸 **Особенности:** Отличная масштабируемость, но требует специфического мышления.

## ⚖️ Плюсы и минусы EDA

| Плюсы | Минусы |
|-------|--------|
| ✅ Слабая связанность компонентов | ❌ Сложность отладки и трассировки |
| ✅ Асинхронность и отказоустойчивость | ❌ Eventual consistency (согласованность в конечном счёте) |
| ✅ Лёгкость добавления новых подписчиков | ❌ Нужно обеспечить идемпотентность |
| ✅ Естественное отражение предметной области | ❌ Сложнее проектировать бизнес-процессы |

---

## 🧠 Когда я выбираю EDA

Я выбираю событийно-ориентированную архитектуру, когда:

1. **Нужна слабая связанность** — компоненты должны жить независимо.
2. **Есть асинхронные процессы** — ответ на запрос не нужен немедленно.
3. **Важна история изменений** — нужен аудит, возможность восстановить состояние.
4. **Система растёт** — добавление новых подписчиков не должно требовать изменений в существующем коде.
5. **Есть несколько потребителей одного события** — например, уведомления, аналитика, логирование.

## 🔗 EDA в сочетании с другими паттернами

| Паттерн | Как сочетается |
|---------|----------------|
| **Микросервисы** | Основной способ коммуникации между сервисами |
| **Saga** | Реализуется через события (хореография) |
| **CQRS** | Команды → события → проекции |
| **Event Sourcing** | Является частным случаем EDA |
| **Clean Architecture** | Use Cases публикуют/подписываются на события |

---

## 🚀 Как использовать этот раздел

1. **Начни с простой EDA** — это быстрый вход в парадигму.
2. **Переходи к Event Sourcing**, когда нужна история изменений и аудит.
3. **Добавь CQRS**, если нагрузка на чтение и запись сильно различается.
4. **Попробуй Actor Model** для высоконагруженных и конкурентных систем.
