# Современные и гибридные подходы

Современная архитектура не сводится к простому выбору между монолитом и микросервисами. Всё чаще решения строятся как **гибридные системы**, где разные части используют разные подходы: часть — как Serverless, часть — как микросервисы, а связь обеспечивается через Service Mesh или брокеры.

В этой директории я описываю ключевые современные подходы, которые помогают строить гибкие, эволюционирующие и экономически эффективные системы.

## 🧠 Что такое «гибридный подход»

Гибридный подход — это не «смесь всего подряд». Это **осознанный выбор разных стилей для разных частей системы**:

- **Serverless** — для событийных и редких задач, где не хочется держать постоянно запущенный сервис.
- **Service Mesh** — для управления сложной сетевой коммуникацией между большим количеством микросервисов.
- **Broker Architecture** — как альтернатива прямому взаимодействию сервисов через API Gateway, когда все общаются через центральную шину.

Гибридная архитектура позволяет:
- Использовать **сильные стороны каждого подхода**.
- **Экономить ресурсы** (Serverless платишь только за время выполнения).
- **Упрощать управление сложностью** (Service Mesh берёт на себя сетевые проблемы).
- **Адаптироваться под разные требования** (разные части системы имеют разную природу).

## 🧩 Основные подходы

### 1. Serverless (FaaS)

Приложение разбивается на отдельные функции, которые выполняются в облачной среде и оплачиваются только за время выполнения.

🔹 **Когда использовать:** 
- Для обработки событий (загрузка файлов, уведомления, триггеры).
- Для редких и нерегулярных задач (резервное копирование, аналитика).
- Для быстрой проверки идей (PoC, MVP).

🔸 **Особенности:**
- Нет управления серверами.
- Масштабируется автоматически.
- Ограниченное время выполнения (обычно до 15 минут).
- Сложно с долгими соединениями и хранилищем состояния.

---

### 2. Service Mesh

Слой инфраструктуры, который управляет коммуникацией между сервисами (маршрутизация, безопасность, наблюдаемость).

🔹 **Когда использовать:**
- Когда у вас десятки или сотни микросервисов.
- Когда нужен единый подход к безопасности (mTLS) и наблюдаемости.
- Когда команды не хотят заботиться о сетевых вопросах в коде.

🔸 **Примеры:** Istio, Linkerd, Consul.
🔸 **Особенности:**
- Добавляет небольшую задержку (latency).
- Требует дополнительных ресурсов (sidecar-контейнеры).
- Упрощает управление сложной сетью.

---

### 3. Broker Architecture

Все сервисы общаются через центральный брокер сообщений, а не напрямую друг с другом.

🔹 **Когда использовать:**
- Когда нужна асинхронная коммуникация.
- Когда много потребителей и поставщиков.
- Когда важна слабая связанность.

🔸 **Особенности:**
- Все запросы проходят через брокер → центральная точка.
- Хорошо масштабируется (брокер может быть кластеризован).
- Поддерживает хореографию и event-driven модели.

---

## ⚖️ Плюсы и минусы гибридных подходов

| Плюсы | Минусы |
|-------|--------|
| ✅ Гибкость — выбираешь лучшее для каждой задачи | ❌ Сложность управления разными подходами |
| ✅ Оптимизация затрат (Serverless — только за выполнение) | ❌ Требует более высокой квалификации команды |
| ✅ Упрощение сложности (Service Mesh берёт сетевые проблемы на себя) | ❌ Дополнительные точки отказа |
| ✅ Лёгкость интеграции с внешними системами (Broker) | ❌ Усложняется мониторинг и отладка |

---

## 🧠 Когда я выбираю гибридные подходы

Я применяю гибридные подходы, когда:

1. **Система большая и разнородная** — разные части имеют разные требования к производительности, задержке, частоте изменений.
2. **Есть жёсткие требования к затратам** — Serverless позволяет платить только за фактическое использование.
3. **Сетевые проблемы становятся заметными** — Service Mesh помогает управлять сложной маршрутизацией и безопасностью.
4. **Нужна асинхронная интеграция** — Broker Architecture даёт слабую связанность и упрощает добавление новых потребителей.

---

## 🚀 Как использовать этот раздел

1. **Оцени, какая часть системы требует быстрого масштабирования и низких затрат** — рассмотри Serverless.
2. **Проверь, не становится ли сеть узким местом** — добавь Service Mesh.
3. **Если требуется асинхронность и слабая связанность** — используй Broker Architecture.
4. **Комбинируй подходы** в зависимости от контекста.
