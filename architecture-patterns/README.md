## 🧠 Как выбирать архитектурный паттерн

Выбор архитектуры — это не вопрос «что модно», а вопрос контекста. Вот несколько ключевых факторов, которые я учитываю:

| Фактор | Вопрос к себе |
|--------|---------------|
| **Размер команды** | Сколько человек будет разрабатывать систему? |
| **Сложность домена** | Насколько сложна предметная область? |
| **Требования к масштабированию** | Как система будет расти? |
| **Необходимость в независимых релизах** | Нужно ли выпускать части системы отдельно? |
| **Бюджет и время** | Какие ресурсы доступны на старте? |
| **Команда и её опыт** | Каков уровень команды? |

## 🏗️ Связь с другими уровнями репозитория

Архитектурные паттерны — это первый уровень абстракции. Они определяют, как будет организована система в целом. Внутри каждого паттерна я использую:

- **Стили организации кода** (`../code-styles/`) — как структурировать код внутри системы.
- **Примеры реализации паттернов** (`../patterns-examples/`) — конкретные решения на Go для частых задач.

---

## 📌 Принципы, которые я соблюдаю

1. **Не догматизм, а прагматизм** — выбор архитектуры зависит от контекста, а не от «моды».
2. **Эволюционность** — архитектура должна уметь меняться вместе с системой.
3. **Документированность** — каждое важное решение фиксируется в ADR.
4. **Чёткие границы** — модули должны быть слабо связаны и сильно связны внутри.

## 💡 Как это использовать

1. **Изучи README.md каждой категории**, чтобы понять, когда применять тот или иной подход.
2. **Сравни разные варианты** с учётом контекста твоего проекта.
3. **Возьми шаблон** из соответствующей папки и адаптируй под свои нужды.
4. **Задокументируй свои решения** с помощью ADR (см. `../docs/architecture/adr/`).

## 🚀 Рекомендация по изучению

Если ты только начинаешь знакомиться с архитектурными паттернами, я рекомендую двигаться так:

1. **Монолитная архитектура** — понять базовые принципы.

3. **Микросервисы** — изучить, когда и как их применять.
4. **Event-Driven и распределённые системы** — для сложных и масштабируемых проектов.
