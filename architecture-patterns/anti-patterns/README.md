# Анти-паттерны в архитектуре

Анти-паттерны — это **повторяющиеся решения проблем, которые кажутся правильными на первый взгляд, но на практике ведут к ухудшению системы и росту технического долга**. Они возникают из-за нехватки опыта, давления сроков или непонимания контекста.

В этой директории я описываю **самые распространённые архитектурные анти-паттерны**, чтобы ты мог их распознавать и избегать.

## 🧠 Почему анти-паттерны опасны

Анти-паттерны не очевидны на старте. Они часто возникают как «быстрое решение» или «проверенный подход», но со временем:

- **Увеличивают сложность** системы.
- **Замедляют разработку** и внедрение изменений.
- **Увеличивают технический долг**.
- **Снижают мотивацию** команды.
- **Мешают масштабированию** и эволюции.

## 🧩 Основные анти-паттерны

### 1. Distributed Monolith (Распределённый монолит)

**Симптомы:**
- Сервисы разбиты по логике, но используют общую БД.
- Изменение в одном сервисе требует изменений в других.
- Все сервисы деплоятся вместе (или с жёсткими зависимостями).
- Нет независимых релизных циклов.

**Причина:** Имитация микросервисов без реальной автономии.

**Решение:**
- Разделить БД по сервисам (или по схемам).
- Сделать API чёткими и стабильными.
- Внедрить события для обмена данными.

---

### 2. Big Ball of Mud (Большой ком грязи)

**Симптомы:**
- Нет чёткой структуры.
- Код перемешан: бизнес-логика, транспорт, работа с БД — всё в одном слое.
- Трудно понять, где что находится.
- Любое изменение затрагивает множество мест.

**Причина:** Отсутствие архитектурного контроля, быстрый старт без плана.

**Решение:**
- Внедрить слои (Clean Architecture, DDD).
- Рефакторить поэтапно.
- Внедрить архитектурные тесты (фитнес-функции).

---

### 3. Stovepipe System (Система-изолированная труба)

**Симптомы:**
- Каждая команда строит свою «трубу» (свой стек, свои данные).
- Интеграции нет, данные дублируются.
- Нет общего видения архитектуры.

**Причина:** Децентрализация без стратегии интеграции.

**Решение:**
- Внедрить общую шину данных (events, API Gateway).
- Создать сервисный каталог.
- Ввести стандарты для API и обмена данными.

---

### 4. Vendor Lock-In (Привязка к вендору)

**Симптомы:**
- Использование проприетарных сервисов без возможности замены.
- Сложность миграции на другую платформу.
- Высокая зависимость от одного облачного провайдера.

**Причина:** Выбор удобства над гибкостью.

**Решение:**
- Использовать абстракции (интерфейсы).
- Избегать вендор-специфичных фич без крайней необходимости.
- Планировать мультиоблачную стратегию.

---

### 5. Golden Hammer (Золотой молоток)

**Симптомы:**
- Один и тот же инструмент используется для всех задач (например, Kafka для всего).
- Решения принимаются на основе «нам это знакомо», а не «это подходит».

**Причина:** Привычка и страх перед новым.

**Решение:**
- Оценивать альтернативы по критериям (нагрузка, latency, сложность).
- Проводить PoC для новых технологий.
- Принимать решения на основе данных, а не эмоций.

---

### 6. Analysis Paralysis (Аналитический паралич)

**Симптомы:**
- Бесконечное обсуждение решений без реализации.
- Проект не двигается вперёд.
- Все боятся ошибиться.

**Причина:** Страх ошибки, перфекционизм.

**Решение:**
- Внедрить Agile-итерации.
- Принимать решения с возможностью отката.
- Использовать «защищённые эксперименты» (feature flags).

---

### 7. Cargo Cult (Карго-культ)

**Симптомы:**
- Копирование паттернов из чужих проектов без понимания контекста.
- Использование «модных» решений (Kubernetes, Kafka) потому что «все так делают».

**Причина:** Желание быть как все, непонимание причин.

**Решение:**
- Изучать «почему», а не только «что».
- Проверять решения на своей нагрузке.
- Не бояться отказываться от «модных» технологий, если они не подходят.

## 🧠 Как распознавать анти-паттерны

| Вопросы, чтобы заметить анти-паттерн | Что проверять |
|--------------------------------------|---------------|
| Зависит ли изменение одного сервиса от изменения другого? | Связанность (coupling) |
| Есть ли у нас чёткие границы между модулями? | Автономность |
| Знаем ли мы, почему используем ту или иную технологию? | Осознанность выбора |
| Можем ли мы развернуть один сервис без развёртывания других? | Независимость релизов |
| Как быстро мы можем откатить изменение? | Гибкость |

---

## 🚀 Как выходить из анти-паттернов

1. **Признать проблему** — анти-паттерн существует и его нужно решать.
2. **Не пытаться исправить всё сразу** — выделить критичные точки.
3. **Внедрять изменения постепенно** (strangler pattern).
4. **Документировать решения и их причины** (ADR).
5. **Автоматизировать проверки** — тестировать архитектурные принципы.
