# Распределённые системы

Распределённая система — это система, состоящая из нескольких независимых узлов (процессов, сервисов, контейнеров), которые взаимодействуют через сеть для достижения общей цели. В отличие от микросервисов (которые фокусируются на архитектурном стиле), распределённые системы охватывают широкий спектр подходов: от кластерных БД до peer-to-peer сетей.

В этой директории я описываю ключевые паттерны распределённых систем, которые помогают строить масштабируемые, отказоустойчивые и производительные решения.

## 🧠 Что такое распределённая система

Ключевые характеристики любой распределённой системы:

- **Множество узлов** — система состоит из нескольких процессов, работающих на разных машинах (или в разных контейнерах).
- **Сетевое взаимодействие** — узлы обмениваются сообщениями через сеть.
- **Отсутствие общей памяти** — каждый узел имеет свою память.
- **Конкурентность** — узлы работают параллельно и асинхронно.
- **Частичные отказы** — часть узлов может выйти из строя, но система должна продолжать работать.

## 🧩 Основные паттерны распределённых систем

### 1. Peer-to-Peer (P2P)

Все узлы равны, нет центрального координатора. Каждый узел может выступать и как клиент, и как сервер.

🔹 **Когда использовать:** Когда важна децентрализация, отказоустойчивость, масштабируемость (например, блокчейн, файловые сети).

🔸 **Особенности:** Сложность координации, согласованности, обнаружения узлов.

---

### 2. Leader-Follower

Один узел — лидер — принимает все решения и управляет репликами. При отказе лидера выбирается новый.

🔹 **Когда использовать:** Для систем с жёсткими требованиями к консистентности (базы данных, кластеры).

🔸 **Особенности:** Единая точка отказа (на время выборов), но простота согласования.

---

### 3. Sharding

Данные разделяются по ключу (хэш, диапазон) между разными узлами. Каждый узел хранит только свою часть данных.

🔹 **Когда использовать:** Когда данные слишком велики для одного узла, нужна линейная масштабируемость.

🔸 **Особенности:** Сложность перераспределения при добавлении узлов, сложность кросс-шардовых запросов.

---

### 4. Space-Based Architecture

Данные хранятся в распределённой памяти, доступной всем узлам (например, Hazelcast, Redis Cluster).

🔹 **Когда использовать:** Когда нужен быстрый доступ к общим данным в реальном времени (кэши, сессии, распределённые счётчики).

🔸 **Особенности:** Ограниченная ёмкость, eventual consistency.

## ⚖️ Общие принципы распределённых систем

| Принцип | Описание |
|---------|----------|
| **CAP-теорема** | Из трёх свойств (Consistency, Availability, Partition Tolerance) можно гарантировать только два. |
| **Eventually Consistency** | Данные в конечном счёте становятся согласованными, но не мгновенно. |
| **Idempotency** | Операции должны быть безопасны при повторении. |
| **Retry & Backoff** | При сбоях повторять попытки с экспоненциальной задержкой. |
| **Circuit Breaker** | Защита от каскадных отказов. |
| **Observability** | Мониторинг, логирование, трассировка — обязательны. |

---

## 🧠 Когда я выбираю распределённую архитектуру

Я выбираю подходы из этой категории, когда:

- **Данные слишком велики** для одной машины (использую Sharding).
- **Нужна высокая доступность** и отказоустойчивость (Leader-Follower).
- **Система должна работать без единой точки отказа** (P2P).
- **Необходим ультра-быстрый доступ к общим данным** (Space-Based).

---

## 🔗 Связь с другими паттернами

- **Микросервисы** часто используют Sharding, Leader-Follower для своих БД.
- **Event-Driven** хорошо сочетается с Peer-to-Peer и Space-Based через распределённые очереди и кэши.
- **Saga** часто полагается на распределённую координацию (Leader-Follower) или децентрализованную (P2P).

---

## 🚀 Как использовать этот раздел

1. **Оцени масштаб данных** — если данные не влезают на один узел, рассмотри Sharding.
2. **Проверь требования к доступности** — если нужна высокая доступность, выбери Leader-Follower.
3. **Рассмотри P2P**, если система должна работать без центрального координатора (например, блокчейн или mesh-сети).
4. **Используй Space-Based**, если нужен сверхбыстрый доступ к данным (например, для кэшей или состояний сессий).
