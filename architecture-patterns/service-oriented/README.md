# Сервис-ориентированная архитектура (SOA)

SOA — это подход к проектированию систем, при котором функциональность разбивается на **сервисы**, предоставляющие чёткие бизнес-интерфейсы. В отличие от микросервисов, SOA часто предполагает **корпоративные масштабы**, **централизованное управление** и **единую шину данных**.

В этой директории я рассматриваю **современную интерпретацию SOA**, которая отказывается от тяжёлых стандартов (XML, SOAP, ESB) в пользу лёгких протоколов (REST, gRPC) и событийной интеграции.

## 🧠 Что такое SOA сегодня

**Классический SOA** (2000-е):
- Тяжёлые протоколы (SOAP/XML)
- Централизованная ESB (Enterprise Service Bus)
- Ориентация на интеграцию корпоративных систем

**Современный SOA** (2020+):
- Лёгкие протоколы (REST, gRPC)
- Децентрализованная или отсутствующая ESB
- Событийно-ориентированное взаимодействие (Kafka, RabbitMQ)
- Акцент на бизнес-капабилити, а не на интеграцию

Современный SOA часто называют **«микросервисами на стероидах»** — он масштабируется на уровне организации, а не одной команды.

---

## ⚖️ SOA vs Микросервисы

| Характеристика | SOA | Микросервисы |
|----------------|-----|---------------|
| **Масштаб** | Корпоративный (межкомандный) | Командный |
| **Управление** | Централизованное | Децентрализованное |
| **Размер сервисов** | Крупные бизнес-сервисы | Мелкие, автономные |
| **Инфраструктура** | Тяжёлая (ESB, оркестрация) | Лёгкая (API Gateway, хореография) |
| **Коммуникация** | SOAP/XML, иногда REST/gRPC | REST/gRPC, событийная шина |
| **Стратегия данных** | Общие БД (интеграция) | Каждый сервис со своей БД |

## 🧩 Ключевые принципы современного SOA

1. **Сервисы ориентированы на бизнес-капабилити**  
   Каждый сервис отражает конкретную бизнес-способность (например, «Управление заказами», «Обработка платежей»).

2. **Слабая связанность (Loose Coupling)**  
   Сервисы общаются через чёткие API, изменения в одном не требуют изменений в другом.

3. **Единая шина данных (но не ESB!) — события**  
   Вместо ESB — брокеры сообщений (Kafka, NATS), которые обеспечивают асинхронную интеграцию.

4. **Сервисный каталог**  
   Централизованное место, где описаны все сервисы, их контракты и владельцы.

5. **Автоматизация CI/CD**  
   Как и в микросервисах, SOA выигрывает от автоматического развёртывания.

## 🏗️ Архитектура современного SOA (принципиальная схема)

```
[ Клиентское приложение ]
         |
    [ API Gateway ]         ← единая точка входа, маршрутизация, авторизация
         |
    [ Сервис A ]  --------→  [ Сервис B ]  ← синхронный вызов (gRPC/REST)
         |                       |
         |  (событие)           (событие)
         ↓                       ↓
    [ Брокер сообщений ]  ← асинхронная шина (Kafka)
         |
    [ Сервис C ]  ← подписчик
```

## ⚙️ Когда я выбираю SOA

Я выбираю **современный SOA**, когда:

- Система **корпоративного уровня** (несколько команд, разные отделы).
- Нужна **интеграция с устаревшими системами** (но я хочу обернуть их в современные API).
- Бизнес-капабилити имеют **чёткие и стабильные границы**.
- Нужна **единая стратегия управления API** и мониторинга.

Если же система автономна и разрабатывается одной командой, я чаще выбираю **модульный монолит** или **микросервисы**.

---

## 📌 Примеры использования SOA в реальной жизни

- **Ритейл**: управление заказами, каталогом товаров, логистикой, платежами.
- **Телеком**: управление абонентами, тарификация, биллинг, сетевые сервисы.
- **Финтех**: обработка транзакций, скоринг, управление рисками, отчётность.

---

## 🚀 Как использовать этот раздел

1. **Оцени, нужна ли тебе SOA** — если система крупная и распределённая, это хороший выбор.
2. **Начни с одного сервиса** — сначала выдели ключевую бизнес-капабилити.
3. **Внедри единую шину** — для асинхронной интеграции между подразделениями.
4. **Поддерживай сервисный каталог** — это ключ к управляемости и масштабированию.
