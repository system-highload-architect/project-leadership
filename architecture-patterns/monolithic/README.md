# Монолитная архитектура

Монолитная архитектура — это подход, при котором всё приложение собирается и развёртывается как единое целое. Несмотря на流行ность микросервисов, монолит остаётся сильным выбором для многих проектов, особенно на ранних стадиях.

В этой директории я описываю три основных варианта монолитной архитектуры и даю рекомендации по их применению.

## 🧠 Когда выбирать монолит

Я выбираю монолитную архитектуру, когда:

1. **Команда небольшая** (до 10 человек) — проще координировать разработку.
2. **Проект стартует с нуля** — быстрый выход на рынок важнее гибкости.
3. **Предметная область ещё не до конца определена** — монолит легче менять на ранних этапах.
4. **Нет жёстких требований к масштабированию** — можно начать с монолита, а потом распилить.

---

## ⚖️ Плюсы и минусы монолита

| Плюсы | Минусы |
|-------|--------|
| ✅ Простота разработки и деплоя | ❌ Сложность масштабирования отдельных частей |
| ✅ Одна кодовая база, один репозиторий | ❌ Риск «большого кома грязи» (Big Ball of Mud) |
| ✅ Лёгкость рефакторинга на ранних этапах | ❌ Ошибка в одном модуле может уронить всю систему |
| ✅ Упрощённые транзакции (ACID) | ❌ Трудности с независимыми релизами частей системы |
| ✅ Меньше операционных затрат | ❌ Может стать узким местом при росте команды и нагрузки |

## 🧩 Варианты монолитной архитектуры

### 1. Классический монолит
Всё в одном коде — бизнес-логика, транспорт, работа с БД, всё вперемешку.

🔹 **Когда использовать:** Для быстрых прототипов, небольших проектов, где не требуется долгосрочная поддержка.

🔸 **Недостаток:** Быстро превращается в «кашу», трудно поддерживать и развивать.

---

### 2. Модульный монолит (Мой основной подход)
Чёткие границы между модулями (DDD), каждый модуль — будущий микросервис. Модули общаются через чёткие интерфейсы или события.

🔹 **Когда использовать:** Для большинства проектов на старте — это идеальный баланс между скоростью и готовностью к росту.

🔸 **Преимущество:** Модули слабо связаны, легко вынести их в микросервисы, когда придёт время.

📌 **Этот подход я использую в своём RMS-проекте.**

---

### 3. Микрокернел (плагинная архитектура)
Ядро системы + подключаемые модули (плагины), которые расширяют функциональность.

🔹 **Когда использовать:** Для систем с чётко выделенным ядром и множеством расширений (например, IDE, браузеры, CMS).

🔸 **Внимание:** В Go этот подход встречается реже, но может быть реализован через интерфейсы и динамическую загрузку.

## 🏗️ Структура модульного монолита (шаблон)

```
modular-monolith/
├── cmd/
│   └── server/
│       └── main.go                 # Точка входа
├── internal/
│   ├── modules/                    # Модули (будущие микросервисы)
│   │   ├── robot-manager/          # Модуль управления роботами
│   │   │   ├── api/                # gRPC/REST интерфейс
│   │   │   ├── service/            # Бизнес-логика
│   │   │   ├── repository/         # Работа с БД
│   │   │   └── events/             # События модуля
│   │   ├── task-dispatcher/        # Модуль распределения задач
│   │   └── location-service/       # Модуль управления локациями
│   ├── shared/                     # Общий код (доменные модели, события, ошибки)
│   │   ├── domain/
│   │   ├── events/
│   │   └── errors/
│   └── infrastructure/             # БД, кэш, брокеры, шина событий
│       ├── postgres/
│       ├── redis/
│       └── eventbus/
├── pkg/                            # Переиспользуемые библиотеки
│   ├── logger/
│   ├── config/
│   └── utils/
├── go.work                         # Для локальной разработки модулей
└── docker-compose.yml              # Локальное окружение
```

## 🔄 Эволюция монолита → микросервисы

Модульный монолит — это идеальный старт для эволюционного перехода к микросервисам:

1. **Начинаем с монолита** с чёткими границами между модулями.
2. **По мере роста нагрузки** выносим модули в отдельные сервисы.
3. **Заменяем in-memory шину** на Kafka (с сохранением интерфейсов).
4. **Каждый модуль становится независимым микросервисом.**

---

## 📌 Когда я отказываюсь от монолита

Я перехожу к микросервисам, когда:

- Команда выросла до 15+ человек.
- Появились требования к независимому масштабированию частей системы.
- Части системы нужно обновлять независимо друг от друга.

---

## 🚀 Как использовать этот раздел

1. **Начни с модульного монолита** — это мой основной выбор для большинства проектов.
2. **Ознакомься со структурой** и адаптируй её под свой проект.
3. **Используй `go.work`** для локальной разработки нескольких модулей.
4. **Следи за границами модулей** — они должны быть чёткими и документированными.
