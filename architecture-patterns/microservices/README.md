# Микросервисная архитектура

Микросервисная архитектура — это подход, при котором приложение строится как набор небольших, автономных сервисов, каждый из которых реализует одну бизнес-способность и может быть развёрнут независимо. Это одна из самых популярных архитектур для сложных, масштабируемых систем, но она требует зрелости в процессах и культуре.

В этой директории я описываю **все ключевые вариации микросервисной архитектуры**, включая их комбинации и особенности реализации.

## 🧠 Что такое микросервисы (и чем они не являются)

**Микросервисы — это не просто «маленькие сервисы»**. Это архитектурный стиль, который требует:

- **Автономности** — каждый сервис можно разрабатывать, тестировать и разворачивать независимо.
- **Децентрализованного управления данными** — у каждого сервиса своя БД (или своя схема).
- **Коммуникации через API** — синхронной (REST/gRPC) и асинхронной (события).
- **Культурной готовности** — команды должны быть автономны и ответственны за свои сервисы.

**Микросервисы — это не серебряная пуля.** Если у вас нет зрелых DevOps-процессов и команды, способной поддерживать распределённую систему, микросервисы могут принести больше проблем, чем пользы.

## 🧩 Все вариации микросервисов

### 1. Базовые микросервисы (basic/)

Каждый сервис решает одну задачу, имеет свою БД и общается с другими через HTTP/gRPC.

🔹 **Когда использовать:** На начальном этапе перехода к микросервисам.

🔸 **Особенности:** Сервисы синхронно вызывают друг друга, что может создавать жёсткую связность и цепочки вызовов.

---

### 2. Микросервисы с API Gateway (api-gateway/)

Внешние клиенты (Web, Mobile) обращаются к единой точке входа — API Gateway, который маршрутизирует запросы к нужным сервисам.

🔹 **Когда использовать:** Когда у системы несколько типов клиентов (Web, Mobile, Third-party).

🔸 **Преимущество:** Единый слой для аутентификации, логирования, rate limiting, агрегации запросов.

---

### 3. Микросервисы с CQRS (cqrs/)

Разделение моделей для чтения (Query) и записи (Command). Это позволяет оптимизировать каждую сторону независимо.

🔹 **Когда использовать:** Когда нагрузка на чтение и запись сильно различается, или нужна сложная аналитика.

🔸 **Особенности:** Часто требует eventual consistency и дополнительных механизмов синхронизации.

---

### 4. Микросервисы с Event Sourcing (event-sourcing/)

Вместо хранения текущего состояния — хранение последовательности событий, которые привели к этому состоянию.

🔹 **Когда использовать:** Когда важна аудит-история, возможность восстановления состояния на любой момент времени.

🔸 **Особенности:** Сложность реализации, большие объёмы данных, но мощная гибкость.

---

### 5. Микросервисы с Saga (saga/)

Управление распределёнными транзакциями через цепочку локальных транзакций с компенсирующими действиями.

🔹 **Когда использовать:** Когда нужно обеспечить консистентность данных между несколькими сервисами.

🔸 **Две модели:** **Хореография** (без центрального координатора) и **Оркестрация** (с центральным координатором).

---

### 6. Микросервисы с Sidecar (sidecar/)

Вспомогательные функции (логирование, метрики, авторизация) выносятся в отдельный контейнер, который работает рядом с основным.

🔹 **Когда использовать:** Когда нужно стандартизировать сквозную функциональность для всех сервисов.

🔸 **Пример:** Istio Envoy, Logstash Sidecar, Prometheus Exporter.

## ⚖️ Микросервисы: плюсы и минусы

| Плюсы | Минусы |
|-------|--------|
| ✅ Независимое масштабирование | ❌ Сложность распределённых транзакций |
| ✅ Независимые релизы | ❌ Сложность отладки и мониторинга |
| ✅ Команды могут выбирать свой стек | ❌ Сетевые задержки (latency) |
| ✅ Устойчивость к сбоям (изоляция) | ❌ Увеличение операционных затрат |
| ✅ Проще понимать код (маленький сервис) | ❌ Нужна зрелая DevOps-культура |

## 🧠 Когда я выбираю микросервисы

Я выбираю микросервисную архитектуру, когда:

- Команда **большая** (15+ разработчиков) и её можно разделить на независимые подкоманды.
- Система имеет **чёткие бизнес-границы** (Bounded Contexts), которые можно вынести в отдельные сервисы.
- Есть **требования к независимому масштабированию** разных частей системы.
- У компании есть **зрелый DevOps** и опыт работы с распределёнными системами.
- Части системы нужно **обновлять независимо** (разные графики релизов).

Если хотя бы одно из этих условий не выполняется, я предпочту **модульный монолит**.

## 🔄 Как микросервисы связаны с другими паттернами

Микросервисы — это **архитектурный стиль**, который может комбинироваться с другими паттернами:

| Паттерн | Как сочетается |
|---------|----------------|
| **Clean Architecture** | Применяется внутри каждого микросервиса |
| **Event-Driven** | Микросервисы общаются через события |
| **CQRS / Event Sourcing** | Используется внутри одного или нескольких сервисов |
| **Saga** | Управляет транзакциями между сервисами |
| **API Gateway** | Единая точка входа для внешних клиентов |
| **Sidecar** | Вынос сквозной функциональности |

---

## 🚀 Как использовать этот раздел

1. **Начни с базовых микросервисов**, чтобы освоить принципы.
2. **Добавь API Gateway**, если у тебя несколько клиентов.
3. **Внедри CQRS или Event Sourcing**, если есть сложные требования к данным.
4. **Используй Saga** для распределённых транзакций.
5. **Примени Sidecar**, если нужна стандартизация сквозных функций.
