# Инженерные практики: RMS (Robot Management System)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Утверждено

---

## 1. Введение

Данный документ описывает **инженерные практики**, которые команда использует при разработке RMS. Они обеспечивают качество кода, предсказуемость релизов, безопасность и эффективность работы. Все члены команды обязаны следовать этим практикам.

Практики охватывают:
- Контроль версий и процесс разработки.
- Непрерывную интеграцию и доставку (CI/CD).
- Code Review и стандарты кода.
- Тестирование.
- Документирование.
- Мониторинг и логирование.
- Управление конфигурацией и секретами.

---

## 2. Контроль версий и процесс разработки

### 2.1 Ветвление (Git Flow / GitHub Flow)
Используем **GitHub Flow** с элементами Git Flow:
- **main** — основная ветка, всегда готовая к релизу.
- **develop** — ветка разработки (опционально, для сложных проектов). В нашем случае используем `main` как основную, а фичи делаем в отдельных ветках.
- **feature/XXX-name** — ветки для новых фичей, багфиксов, экспериментов.
- **hotfix/XXX-name** — для срочных исправлений в продакшене.

### 2.2 Правила именования веток
- `feature/название-задачи` (например, `feature/robot-crud`)
- `fix/описание-бага` (например, `fix/order-status-bug`)
- `hotfix/описание` (например, `hotfix/kafka-connection-leak`)

### 2.3 Коммиты
- Используем [Conventional Commits](https://www.conventionalcommits.org/):
  - `feat: добавлен эндпоинт создания робота`
  - `fix: исправлен баг с дублированием заказов`
  - `docs: обновлён README`
  - `test: добавлены тесты для usecase`
  - `chore: обновлён go.mod`
- Каждый коммит должен быть логически завершённым.
- Коммиты пишем на **русском** (в пределах команды) или английском (для open-source).

### 2.4 Pull Request'ы
- Каждая фича / багфикс должна быть оформлена через Pull Request (PR).
- PR должен содержать:
  - Описание изменений.
  - Ссылку на задачу (Jira / GitHub Issue).
  - Чеклист (пройдены ли тесты, не нарушена ли документация).
- PR должен быть рассмотрен минимум двумя ревьюерами (один из них — Tech Lead).

## 3. CI/CD (Непрерывная интеграция и доставка)

### 3.1 CI (GitHub Actions)
При каждом пуше в любую ветку запускается CI-пайплайн:
1. **Линтинг** — `golangci-lint` для Go, `eslint` для фронтенда.
2. **Тесты** — unit-тесты + интеграционные тесты (с testcontainers).
3. **Сборка** — сборка Docker-образов (без публикации, только проверка).
4. **Проверка миграций** — валидация SQL-миграций.

При пуше в `main` (или `develop`) дополнительно:
- Публикация Docker-образов в реестр (GitHub Container Registry).
- Автоматический деплой на staging-окружение.

### 3.2 CD (Доставка)
- **Staging**: автоматический деплой после каждого успешного CI на `main`.
- **Production**: деплой вручную (или по расписанию) через GitHub Actions с подтверждением (approval).
- Используем стратегию **Rolling Update** или **Blue-Green Deployment** (в зависимости от окружения).

### 3.3 Откат
- В случае проблем на staging/production выполняется откат к предыдущей версии (через `kubectl rollout undo`).
- Время отката — не более 5 минут.

## 4. Code Review

### 4.1 Критерии ревью
- Код соответствует Clean Architecture (слои, зависимости).
- Есть тесты (unit / integration) на новую функциональность.
- Нет дублирования кода (DRY).
- Обработка ошибок — явная, с контекстом.
- API-контракты обновлены (OpenAPI / Protobuf).
- Документация (README, ADR) обновлена при необходимости.
- Нет уязвимостей (зависимости проверены, нет секретов в коде).

### 4.2 Процесс ревью
- PR не мержится без двух аппрувов.
- Ревьюер должен оставить комментарии по коду, указать на сильные и слабые места.
- Автор PR должен ответить на все комментарии и внести исправления.
- После исправлений ревьюер пересматривает изменения.

## 5. Стандарты кода

### 5.1 Go
- Следовать [Effective Go](https://go.dev/doc/effective_go) и [Uber Go Style Guide](https://github.com/uber-go/guide).
- Использовать `go fmt`, `go vet`, `golangci-lint` (с настроенным конфигом).
- Именование:
  - Пакеты — в нижнем регистре, короткие (например, `usecase`, `repository`).
  - Экспортируемые типы — с большой буквы, неэкспортируемые — с маленькой.
  - Интерфейсы — по возможности с суффиксом `-er` (например, `RobotRepository`).
- Обработка ошибок: всегда проверять ошибки, использовать `fmt.Errorf` с `%w` для цепочки ошибок.

### 5.2 React / TypeScript
- Следовать [Airbnb React Style Guide](https://github.com/airbnb/javascript/tree/master/react).
- Использовать функциональные компоненты и хуки (не классы).
- Типизировать всё (TypeScript).
- Компоненты — в отдельных файлах, экспорт по умолчанию.

### 5.3 SQL
- Имена таблиц и колонок — в нижнем регистре, слова через `_` (snake_case).
- Все миграции пишутся на языке SQL (без ORM).
- Индексы добавляются по необходимости (на внешние ключи, часто используемые поля).

## 6. Тестирование

### 6.1 Виды тестов
- **Unit-тесты** — тестирование изолированных компонентов (Use Cases, Domain). Используем `testing` + `testify` + моки (gomock или ручные).
- **Integration-тесты** — тестирование с реальной БД (testcontainers), брокером (Kafka in-memory / testcontainers).
- **API-тесты** — проверка эндпоинтов через `net/http/httptest` + валидация ответов.
- **E2E-тесты** — для критических сценариев (например, создание заказа через весь стек). Могут быть на Go или на Cypress (для UI).
- **Нагрузочное тестирование** — с помощью K6 / Locust (на этапе релиза).

### 6.2 Покрытие кода
- Минимальное покрытие для нового кода — 80% (по строкам).
- Проверяется в CI через `go test -cover`.
- В случае падения покрытия — PR не мержится.

### 6.3 Идемпотентность тестов
- Тесты должны быть идемпотентными (можно запускать несколько раз без побочных эффектов).
- Использовать транзакционные тесты (rollback после каждого теста) для интеграционных тестов.

## 7. Документирование

### 7.1 Код
- Публичные функции, методы, типы — комментируются (godoc).
- Внутренние сложные участки — комментарии на русском (для понимания командой).
- Используем `//+` для генерации кода (например, `//go:generate`).

### 7.2 API
- OpenAPI-спецификация (`openapi.yaml`) поддерживается в актуальном состоянии.
- Swagger UI доступен по `/swagger/` в dev/staging.
- Для gRPC — комментарии в `.proto` файлах.

### 7.3 Архитектура
- ADR пишутся для каждого значимого решения.
- Диаграммы (C4, ER, UML) обновляются при изменении архитектуры.
- README в каждом модуле описывает его назначение.

## 8. Мониторинг и логирование

### 8.1 Логи
- Все логи — в JSON-формате (структурированные).
- Уровни: `DEBUG` (локально), `INFO` (staging/prod), `WARN`, `ERROR`.
- Логи содержат `trace_id` для трассировки запросов.
- Логи отправляются в ELK / Loki и доступны для поиска.

### 8.2 Метрики
- Prometheus: собираем метрики HTTP-запросов, ошибок, времени ответа, использования ресурсов.
- Графана: дашборды для мониторинга производительности и доступности.

### 8.3 Алерты
- Настроены в Prometheus (Alertmanager) с отправкой в Slack / Email.
- Пороговые значения: ошибки > 1%, задержка > 500 мс, недоступность сервиса.

## 9. Управление конфигурацией и секретами

### 9.1 Конфигурация
- Используем переменные окружения (12-factor app).
- Для разных окружений — разные `.env` файлы (не хранятся в репозитории).
- Конфигурация загружается при старте приложения.

### 9.2 Секреты
- Секреты (пароли БД, токены) хранятся в **HashiCorp Vault** или **Kubernetes Secrets**.
- Доступ к секретам — только у сервисов, через ServiceAccount.
- Никаких секретов в коде или `.env` в репозитории.
