# Техническое задание: Robot Management System (RMS)

**Версия:** 1.0 (MVP)  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

### 1.1 Цель документа
Данный документ описывает функциональные и нефункциональные требования к системе управления роботами (RMS), определяет архитектурные ограничения и критерии приемки. Он служит основой для разработки, тестирования и приемки MVP.

### 1.2 Краткое описание системы
RMS — это платформа для управления роботами, задачами и заказами. Система позволяет операторам создавать роботов, назначать им задачи, объединять задачи в заказы и отслеживать выполнение в реальном времени. Взаимодействие осуществляется через REST API, gRPC и Web-интерфейс.

### 1.3 Целевая аудитория
- Операторы (управление роботами и задачами).
- Администраторы (настройка системы, мониторинг).
- Системные интеграторы (подключение внешних систем через API).

---

## 2. Общие требования

### 2.1 Бизнес-цели
- Автоматизация управления роботами и задачами.
- Сокращение времени на рутинные операции (назначение, отслеживание).
- Прозрачность выполнения заказов в реальном времени.
- Масштабируемость под рост количества роботов (до 1000+).

### 2.2 Ограничения и допущения
- Первая версия поддерживает только синхронные команды и асинхронные события (Kafka).
- Роботы взаимодействуют через HTTP/gRPC (физические интерфейсы не входят в MVP).
- Система развёртывается в облаке (AWS/GCP) с использованием Kubernetes.
- Язык разработки — Go, фронтенд — React + TypeScript.

## 3. Функциональные требования

### 3.1 Управление роботами
| ID | Функция | Описание |
|----|---------|----------|
| FR-01 | Создание робота | Оператор создаёт робота с именем, типом, начальной локацией. Статус по умолчанию — `idle`. |
| FR-02 | Редактирование робота | Изменение имени, типа, локации. |
| FR-03 | Удаление робота | Удаление робота из системы (только если нет активных задач). |
| FR-04 | Просмотр списка роботов | Фильтрация по статусу, типу, локации. |
| FR-05 | Изменение статуса | Оператор вручную меняет статус (`busy`, `maintenance`, `offline`). |

### 3.2 Управление задачами
| ID | Функция | Описание |
|----|---------|----------|
| FR-06 | Создание задачи | Оператор создаёт задачу с описанием, приоритетом, локацией. Статус — `pending`. |
| FR-07 | Назначение задачи роботу | Оператор назначает существующую задачу роботу. Статус задачи → `in_progress`, статус робота → `busy`. |
| FR-08 | Завершение задачи | Оператор отмечает задачу выполненной (`completed`) или проваленной (`failed`). Статус робота → `idle`. |
| FR-09 | Просмотр задач | Фильтрация по статусу, роботу, заказу. |
| FR-10 | Отмена задачи | Отмена задачи (возврат в `pending` или удаление). |

### 3.3 Управление заказами
| ID | Функция | Описание |
|----|---------|----------|
| FR-11 | Создание заказа | Оператор создаёт заказ (контейнер для одной или нескольких задач). |
| FR-12 | Добавление задачи в заказ | Связывает существующую задачу с заказом. |
| FR-13 | Выполнение заказа | Автоматически отслеживает статус всех задач; при завершении всех — статус заказа → `completed`. |
| FR-14 | Отмена заказа | Отмена заказа (отмена всех задач в нём). |

### 3.4 Управление локациями
| ID | Функция | Описание |
|----|---------|----------|
| FR-15 | Создание локации | Добавление локации с названием, адресом, вместимостью. |
| FR-16 | Редактирование локации | Изменение параметров локации. |
| FR-17 | Удаление локации | Удаление, если нет роботов или задач, привязанных к ней. |
| FR-18 | Просмотр локаций | Список всех локаций с количеством роботов. |

### 3.5 Событийная модель (системная)
| ID | Функция | Описание |
|----|---------|----------|
| FR-19 | Публикация событий | Каждое изменение статуса (создание, изменение, завершение) публикуется в Kafka. |
| FR-20 | Подписка на события | Внешние системы могут подписываться на события для интеграции. |
| FR-21 | Аудит событий | Все события сохраняются для возможности восстановления состояния. |

## 4. Нефункциональные требования

### 4.1 Производительность и масштабируемость
| ID | Требование | Целевое значение |
|----|------------|------------------|
| NFR-01 | Время ответа API (p95) | < 200 мс для чтения, < 500 мс для записи |
| NFR-02 | Пропускная способность | 1000 запросов/сек (чтение), 500 запросов/сек (запись) |
| NFR-03 | Поддержка роботов | До 1000 активных роботов |
| NFR-04 | Масштабирование | Горизонтальное масштабирование через Kubernetes |

### 4.2 Надёжность и доступность
| ID | Требование | Целевое значение |
|----|------------|------------------|
| NFR-05 | Доступность системы | 99.9% (SLO) |
| NFR-06 | Время восстановления (RTO) | 1 час |
| NFR-07 | Потери данных (RPO) | 15 минут |
| NFR-08 | Отказоустойчивость | Репликация БД, Kafka-кластер, Kubernetes-кластер |

### 4.3 Безопасность
| ID | Требование | Описание |
|----|------------|----------|
| NFR-09 | Аутентификация | JWT (Access + Refresh токены) |
| NFR-10 | Авторизация | RBAC (роли: оператор, администратор, системный) |
| NFR-11 | Шифрование данных | TLS для всех внешних соединений |
| NFR-12 | Безопасность API | Rate limiting, валидация входных данных, защита от инъекций |

### 4.4 Наблюдаемость
| ID | Требование | Описание |
|----|------------|----------|
| NFR-13 | Логирование | Структурированные логи (JSON) с уровнями (debug, info, warn, error) |
| NFR-14 | Метрики | Prometheus: время ответа, ошибки, количество запросов |
| NFR-15 | Трассировка | Распределённая трассировка (Jaeger/OpenTelemetry) для микросервисной архитектуры |
| NFR-16 | Алерты | Настройка алертов в Slack/email при превышении SLO |

### 4.5 Технические ограничения
- Язык бэкенда: Go (версия 1.21+).
- База данных: PostgreSQL (версия 15+).
- Брокер сообщений: Kafka (версия 3.x).
- Кэш: Redis.
- Контейнеризация: Docker + Kubernetes.
- CI/CD: GitHub Actions.

## 5. Критерии приемки

| № | Критерий | Способ проверки |
|---|----------|-----------------|
| 1 | Все эндпоинты API (OpenAPI) реализованы и работают | Postman-тесты, интеграционные тесты |
| 2 | События публикуются в Kafka при изменениях | Проверка наличия событий в топиках |
| 3 | UI позволяет выполнять все операции (создание, редактирование, удаление) | Ручное тестирование, E2E-тесты |
| 4 | Время ответа API < 200 мс при нагрузке 1000 запросов/сек | Нагрузочное тестирование (K6) |
| 5 | Система восстанавливается после сбоя за < 1 часа | Chaos-тестирование (убийство pod'ов) |
| 6 | Документация API и архитектурные диаграммы актуальны | Проверка reviewers |

---

## 6. Технические ограничения

- Все API версионируются (например, `/v1/`).
- Имена сущностей и полей — на английском языке (совместимость с кодом и документацией).
- Все изменения проходят code review и CI (автоматические проверки стиля, тесты, линтеры).
- Секреты и конфигурация хранятся в HashiCorp Vault / Kubernetes Secrets.

### Приложение B: Глоссарий
- **Робот** — исполнитель, который может выполнять задачи.
- **Задача** — единица работы, назначаемая роботу.
- **Заказ** — контейнер для одной или нескольких задач.
- **Локация** — место, где роботы работают и задачи выполняются.
- **Событие** — факт изменения состояния, публикуемый в шину.
