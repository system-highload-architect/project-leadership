# 📋 Проектные документы

**Эта директория содержит документы, которые обеспечивают управление проектом, планирование и организацию работы команды.**  
Здесь собраны артефакты, помогающие ответить на вопросы:
- **Что именно мы делаем?** (техническое задание, требования)
- **Кто, что и когда делает?** (план спринта, распределение задач)
- **Как мы работаем?** (инженерные практики, CI/CD, code review)

Эти документы — **мостик между стратегией (зачем) и реализацией (как)**. Они синхронизируют команду, менеджеров и стейкхолдеров относительно текущего состояния проекта и ближайших планов.

---

## 🧠 Принципы проектного управления

1. **Прозрачность.** Все документы доступны всей команде, статусы задач видны каждому.
2. **Гибкость.** Планы могут меняться, но изменения должны быть документированы и согласованы.
3. **Регулярность.** Спринты, встречи и ретроспективы проводятся по расписанию.
4. **Качество.** Инженерные практики — это не «бюрократия», а инструменты для поддержания высокого качества кода и продукта.
5. **Связность.** Проектные документы должны быть связаны со стратегическими (`../strategic/`) и архитектурными (`../architecture/`) документами.

## 📄 Описание документов

### 1. Техническое задание (`technical-spec.md`)

**Что содержит:**
- **Введение** — цель документа, краткое описание проекта.
- **Общие требования** — бизнес-цели, пользователи, ограничения.
- **Функциональные требования** — детальное описание того, что система должна делать (use cases, сценарии).
- **Нефункциональные требования** — производительность, безопасность, надёжность, масштабируемость.
- **Технические ограничения** — технологии, платформы, совместимость.
- **Критерии приемки** — как будет проверяться готовность.
- **Приложения** — ссылки на связанные документы (ADR, диаграммы, API-контракты).

**Когда использовать:**  
- На старте проекта или крупной фичи.
- Для согласования с заказчиком и стейкхолдерами.
- Как основу для планирования спринтов и разработки.

**Связь:**  
- Базируется на `product-vision.md` и `roadmap.md` из `strategic/`.
- Использует архитектурные решения из `architecture/adr/`.

---

### 2. План спринта (`sprint-planning.md`)

**Что содержит:**
- **Цель спринта** — что должно быть достигнуто.
- **Бэклог спринта** — список задач (user stories, баги, технические улучшения) с приоритетами и оценками.
- **Распределение задач** — кто за какую задачу отвечает.
- **Результаты предыдущего спринта** — что сделано, что пошло не так (ретроспектива).
- **Риски и блокеры** — что может помешать, как смягчить.

**Когда использовать:**  
- В начале каждого спринта (обычно раз в 1–2 недели).
- Для синхронизации команды и менеджеров.

**Связь:**  
- Базируется на `technical-spec.md` и `roadmap.md`.
- Учитывает ресурсы из `budget-resources.md` (`strategic/`).

---

### 3. Инженерные практики (`engineering-practices.md`)

**Что содержит:**
- **Контроль версий** — правила ветвления, коммитов, pull request'ов (Git Flow / GitHub Flow).
- **CI/CD** — автоматические сборки, тестирование, деплой.
- **Code Review** — критерии ревью, процесс проверки кода.
- **Тестирование** — стратегия тестирования (unit, integration, e2e, performance).
- **Документирование** — требования к комментариям, документации API, ADR.
- **Мониторинг и логирование** — стандарты для сбора логов и метрик.
- **Управление конфигурацией** — секреты, переменные окружения.

**Когда использовать:**  
- На старте проекта для установки стандартов.
- При онбординге новых разработчиков.
- Для периодического аудита и улучшения процессов.

**Связь:**  
- Связан с `operations/deployment.md` и `operations/runbook.md`.
- Определяет, как архитектура реализуется в коде и инфраструктуре.

---

## 🚀 Как использовать проектные документы

1. **Начни с `technical-spec.md`** — это основа для всей разработки.
2. **Разбей задачи на спринты** с помощью `sprint-planning.md`.
3. **Следуй `engineering-practices.md`** на протяжении всего проекта.
4. **Обновляй документы по мере изменения требований** и ретроспектив.
5. **Связывай с другими разделами** — стратегическими, архитектурными и операционными.

---

## 📌 Важно

- **Техническое задание — живой документ.** Оно должно обновляться при изменении требований.
- **План спринта** — согласуется с командой на старте спринта и фиксируется.
- **Инженерные практики** — обязательны для всех, их соблюдение контролируется на код-ревью и CI.
