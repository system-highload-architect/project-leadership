# Runbook: RMS (Robot Management System)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

Данный документ содержит **инструкции для операторов и инженеров поддержки** по диагностике и устранению типовых инцидентов в системе RMS. Runbook используется в экстренных ситуациях для минимизации времени простоя.

**Цель:** Обеспечить быстрое и предсказуемое реагирование на сбои, восстановление сервиса и эскалацию при необходимости.

---

## 2. Общие принципы

1. **Безопасность прежде всего** — не выполняйте опасных команд (например, `DROP TABLE`) без необходимости.
2. **Логируйте всё** — фиксируйте свои действия, время, причины и результаты.
3. **Не паникуйте** — следуйте инструкциям, не принимайте необдуманных решений.
4. **Эскалируйте своевременно** — если не уверены, что делаете, или не можете восстановить сервис за 15 минут — зовите Tech Lead / DevOps.
5. **Сохраняйте данные для постмортема** — логи, дампы, дашборды — всё пригодится для анализа.

## 3. Инциденты и сценарии

### 3.1 Сервис недоступен (HTTP 5xx / таймауты)

#### Симптомы
- Ошибки 500, 502, 503, 504.
- Таймауты при вызовах API.
- UI не загружается или выдаёт ошибки.

#### Возможные причины
1. Сбой в одном из pod'ов (crash loop).
2. Перегрузка БД / Kafka / Redis.
3. Проблемы сети / DNS.
4. Утечка памяти или ресурсов.

#### Действия
1. **Проверьте статус pod'ов:**
   ```bash
   kubectl get pods -n rms
   kubectl describe pod <pod-name> -n rms
   kubectl logs <pod-name> -n rms --tail=100
   ```

2. **Проверьте здоровье API:**
   ```bash
   curl http://api.rms.local/health
   ```

3. **Проверьте ресурсы (CPU, память):**
   ```bash
   kubectl top pods -n rms
   ```

4. **Если pod в статусе `CrashLoopBackOff`:**
   - Посмотрите логи (`kubectl logs`).
   - Проверьте, нет ли ошибок подключения к БД / Kafka / Redis.
   - Перезапустите pod: `kubectl delete pod <pod-name> -n rms`.

5. **Если проблема с БД:**
   - Проверьте доступность БД (`kubectl exec` внутрь pod'а и выполните `psql`).
   - Проверьте количество соединений:
     ```sql
     SELECT count(*) FROM pg_stat_activity;
     ```
   - Проверьте блокировки:
     ```sql
     SELECT * FROM pg_locks WHERE NOT granted;
     ```

6. **Если проблема с Kafka:**
   - Проверьте, что брокеры доступны.
   - Проверьте топики и группы потребителей.
   - Проверьте, нет ли накопления сообщений (lag).

#### Эскалация
Если не удалось восстановить за 15 минут — позвать Tech Lead / DevOps.

---

### 3.2 Высокая задержка (latency > 500ms)

#### Симптомы
- UI тормозит, запросы долго выполняются.
- Метрики показывают высокий p95 / p99.

#### Возможные причины
1. Медленные запросы к БД (отсутствие индексов, полное сканирование).
2. Перегрузка CPU / памяти в сервисах.
3. Сетевые задержки.
4. Медленная работа Kafka (отставание потребителей).

#### Действия
1. **Проверьте метрики в Grafana:**
   - Время ответа API.
   - Время выполнения SQL-запросов.
   - Использование CPU / памяти.

2. **Найдите медленные запросы:**
   - Включите логи медленных запросов в PostgreSQL:
     ```sql
     SET log_min_duration_statement = 200;
     ```
   - Проверьте `pg_stat_statements` для анализа запросов.

3. **Проверьте индексы:**
   ```sql
   SELECT schemaname, tablename, indexname FROM pg_indexes WHERE schemaname = 'public';
   ```

4. **Проверьте количество соединений и пулы:**
   - Увеличьте pool size в конфигурации приложения.
   - Проверьте, не закрываются ли соединения слишком часто.

5. **Проверьте Kafka lag:**
   ```bash
   kafka-consumer-groups --bootstrap-server kafka:9092 --group <group> --describe
   ```
   Если lag растёт — увеличьте количество потребителей или партиций.

#### Эскалация
Если проблема связана с БД — вызвать DBA или DevOps. Если с кодом — позвать разработчика, ответственного за модуль.

---

### 3.3 Ошибки аутентификации / авторизации (HTTP 401 / 403)

#### Симптомы
- Пользователи не могут войти.
- API возвращает 401 Unauthorized / 403 Forbidden.
- JWT токены не принимаются.

#### Возможные причины
1. Истёк JWT секрет (изменился `JWT_SECRET`).
2. Проблемы с OAuth / SSO (если используется).
3. Истекли токены (не обновляются).
4. Роли и права изменились, но не синхронизировались.

#### Действия
1. **Проверьте конфигурацию JWT:**
   - Сравните `JWT_SECRET` в ConfigMap / Secrets с ожидаемым.
   - Проверьте срок жизни токенов (`JWT_EXPIRE`).

2. **Проверьте логи аутентификации:**
   ```bash
   kubectl logs <backend-pod> -n rms | grep -i auth
   ```

3. **Временно отключите аутентификацию (только для диагностики, не в production!):**
   - Используйте тестовый токен или bypass-режим (если настроен).

4. **Проверьте интеграцию с внешними системами:**
   - Если используется Keycloak / Auth0 — проверьте их статус.

#### Эскалация
Вызвать разработчика, ответственного за auth-модуль, или DevOps.

---

### 3.4 Ошибки при работе с Kafka (события не доставляются)

#### Симптомы
- События не публикуются.
- Подписчики не получают события.
- В логах ошибки `failed to produce message`.

#### Возможные причины
1. Kafka недоступна (сбой брокера).
2. Топик не существует или удалён.
3. Нет свободного места на диске.
4. Ошибки сериализации / десериализации.

#### Действия
1. **Проверьте статус Kafka:**
   ```bash
   kubectl get pods -n kafka
   ```

2. **Проверьте топики:**
   ```bash
   kafka-topics --bootstrap-server kafka:9092 --list
   ```

3. **Проверьте место на диске:**
   ```bash
   df -h /var/lib/kafka/data
   ```

4. **Проверьте ошибки в логах продюсера/консьюмера:**
   ```bash
   kubectl logs <backend-pod> -n rms | grep -i kafka
   ```

5. **Перезапустите продюсера / консьюмера:**
   ```bash
   kubectl delete pod <pod-name> -n rms
   ```

#### Эскалация
Вызвать DevOps или администратора Kafka.

### 3.5 Ошибки БД (соединение потеряно, нарушения целостности)

#### Симптомы
- В логах `connection refused`, `too many connections`, `deadlock detected`.
- Операции зависают или падают.

#### Возможные причины
1. БД недоступна (сбой реплики, перезагрузка).
2. Исчерпаны соединения (max_connections).
3. Deadlock (взаимная блокировка транзакций).
4. Диск заполнен.

#### Действия
1. **Проверьте доступность БД:**
   ```bash
   kubectl exec -it <backend-pod> -n rms -- psql -h postgres-service -U rms_user -d rms_db -c "SELECT 1"
   ```

2. **Проверьте количество соединений:**
   ```sql
   SELECT count(*) FROM pg_stat_activity;
   SELECT count(*) FROM pg_stat_activity WHERE state = 'idle';
   ```

3. **Завершите висячие соединения (если нужно):**
   ```sql
   SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle' AND pid <> pg_backend_pid();
   ```

4. **Проверьте дисковое пространство:**
   ```bash
   df -h /var/lib/postgresql/data
   ```

5. **Проверьте наличие deadlock'ов:**
   ```sql
   SELECT * FROM pg_locks WHERE NOT granted;
   ```

#### Эскалация
Если не удаётся восстановить — вызвать DBA или DevOps.

## 4. Эскалация

| Уровень | Контакт | Условия эскалации |
|---------|---------|-------------------|
| **Уровень 1** | Оператор / Инженер поддержки | Первичная диагностика, базовые действия (перезапуск pod'ов) |

## 5. Постмортем (Post-Mortem)

После каждого инцидента проводится анализ:

1. **Что произошло?** — краткое описание инцидента.
2. **Когда произошло?** — время начала и окончания.
3. **Причина** — корневая причина (root cause).
4. **Действия** — что делали для восстановления.
5. **Уроки** — что можно улучшить (мониторинг, документация, код, инфраструктура).
6. **Action items** — конкретные задачи для предотвращения повторения.
