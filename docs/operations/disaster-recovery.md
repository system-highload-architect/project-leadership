# План восстановления после катастроф (DRP): RMS

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

Данный документ описывает **план действий** при серьёзных сбоях, которые могут привести к длительной недоступности системы, потере данных или другим критическим последствиям.

**Цель:** Минимизировать время простоя (RTO) и потери данных (RPO), обеспечить восстановление системы в предсказуемые сроки.

---

## 2. Ключевые показатели (RTO / RPO)

| Показатель | Целевое значение | Описание |
|------------|------------------|----------|
| **RTO** (Recovery Time Objective) | **1 час** | Максимальное время восстановления сервиса после катастрофы |
| **RPO** (Recovery Point Objective) | **15 минут** | Максимально допустимые потери данных (время с последнего бэкапа) |

---

## 3. Сценарии катастроф

| Сценарий | Описание | Влияние |
|----------|----------|---------|
| **Отказ всего кластера Kubernetes** | Полная недоступность всех сервисов | Критическое |
| **Отказ БД (PostgreSQL)** | Потеря основного инстанса, недоступность данных | Критическое |
| **Отказ брокера Kafka** | События не доставляются, потеря аудита | Высокое |
| **Атака / компрометация** | Взлом, утечка данных, шифрование | Критическое |
| **Сбой облачного провайдера (регион)** | Недоступность всей инфраструктуры в регионе | Критическое |
| **Человеческая ошибка** | Удаление данных, неправильная конфигурация | Высокое / Критическое |

---

## 4. Стратегии восстановления

### 4.1 Отказ кластера Kubernetes

**Действия:**
1. Восстановить кластер из инфраструктурного кода (Terraform / GitOps).
2. Применить манифесты Kubernetes из репозитория (`kubectl apply -f k8s/`).
3. Восстановить данные из бэкапов (см. п. 5).
4. Проверить работоспособность health-эндпоинтов.
5. Переключить DNS на новый кластер.

### 4.2 Отказ БД (PostgreSQL)

**Действия:**
1. Если есть реплика — повысить её до мастера (`promote`).
2. Если нет реплики — восстановить из последнего бэкапа (pg_dump / WAL).
3. Применить WAL-логи с момента последнего бэкапа (Point-in-Time Recovery).
4. Обновить строку подключения в ConfigMap / Secrets.
5. Перезапустить backend-поды для переподключения.

### 4.3 Отказ брокера Kafka

**Действия:**
1. Если кластер Kafka мультиброкерный — переключиться на работающие брокеры.
2. Если все брокеры недоступны — восстановить из бэкапов тем (kafka-dump).
3. Перезапустить продюсеров и консьюмеров.
4. Проверить, что нет потери событий (сравнить offsets).

### 4.4 Атака / компрометация

**Действия:**
1. Изолировать систему (заблокировать сетевой доступ).
2. Сменить все секреты, пароли, токены (Vault / Kubernetes Secrets).
3. Восстановить систему из доверенных бэкапов.
4. Провести аудит безопасности и закрыть уязвимости.
5. Уведомить стейкхолдеров и регуляторов (при необходимости).

### 4.5 Сбой облачного провайдера (регион)

**Действия:**
1. Активировать резервный регион (если настроена мультирегиональность).
2. Переключить DNS на резервный регион (TTL < 5 минут).
3. Проверить работоспособность в резервном регионе.

## 5. Резервное копирование (Backup Strategy)

| Компонент | Частота | Метод | Хранение | Восстановление |
|-----------|---------|-------|----------|----------------|
| **PostgreSQL** | Ежечасно (полный) + WAL (реалтайм) | pg_basebackup + WAL-архивация | S3 / облачный объектный сторедж | Point-in-Time Recovery (до 15 минут) |
| **Kafka** | Ежедневно | kafka-dump / MirrorMaker 2.0 | S3 | Восстановление топиков (оффсеты) |
| **Redis** | Ежечасно | SAVE / BGSAVE | S3 | Восстановление из RDB-файла |
| **Конфигурация** | Каждое изменение | Git (Terraform + Kubernetes manifests) | GitHub | `kubectl apply -f` / Terraform apply |
| **Секреты** | Каждое изменение | HashiCorp Vault (зашифровано) | Vault + бэкап | Восстановить из Vault |

**Резервное копирование проверяется ежемесячно** — проводится тестовое восстановление на отдельном окружении.

## 6. Процедура восстановления (пошагово)

1. **Активировать DRP:** Уведомить всех членов команды и стейкхолдеров.
2. **Диагностика:** Определить масштаб и причину катастрофы.
3. **Восстановление инфраструктуры:** Применить инфраструктурный код (Terraform, Kubernetes manifests).
4. **Восстановление данных:** Восстановить БД, Kafka, Redis из бэкапов.
5. **Проверка:** Запустить smoke-тесты, проверить целостность данных.
6. **Переключение трафика:** Обновить DNS / Ingress на восстановленную систему.
7. **Мониторинг:** Наблюдать за системой, убедиться в стабильности.
8. **Постмортем:** Провести анализ инцидента и обновить DRP.

## 9. Тестирование DRP

- **Полное тестирование:** раз в квартал (симуляция полного отказа).
- **Частичное тестирование:** раз в месяц (восстановление БД из бэкапа).
- **Результаты тестирования** фиксируются и анализируются.
