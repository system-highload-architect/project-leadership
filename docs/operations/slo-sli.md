# SLO/SLI: RMS (Robot Management System)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Утверждено

---

## 1. Введение

Данный документ описывает **Service Level Indicators (SLI)** и **Service Level Objectives (SLO)** для системы RMS. Они определяют, как мы измеряем надёжность, производительность и доступность сервиса, и какие цели ставим перед собой.

SLO и SLI — это основа для принятия решений о масштабировании, улучшении кода и инфраструктуры, а также для коммуникации с бизнесом и стейкхолдерами.

---

## 2. Определения

| Термин | Описание |
|--------|----------|
| **SLI** (Service Level Indicator) | Количественная метрика, измеряющая производительность или надёжность (например, время ответа, частота ошибок, доступность). |
| **SLO** (Service Level Objective) | Целевое значение для SLI (например, 99.9% доступность, 95% запросов < 200 мс). |
| **SLA** (Service Level Agreement) | Юридически обязывающее соглашение с клиентами (обычно на основе SLO). Внутри команды мы используем SLO. |

## 3. SLI (Service Level Indicators)

### 3.1 Доступность (Availability)
**SLI:** Доля успешных запросов к API (HTTP 2xx / 4xx / 5xx).  
**Метод измерения:** Prometheus — `rate(http_requests_total{status!~"5.."}[5m]) / rate(http_requests_total[5m])`.

### 3.2 Задержка (Latency)
**SLI:** Время ответа API (p95, p99) и gRPC-вызовов.  
**Метод измерения:** Prometheus — `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))`.

### 3.3 Частота ошибок (Error Rate)
**SLI:** Доля запросов, завершившихся с ошибкой (HTTP 5xx).  
**Метод измерения:** Prometheus — `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])`.

### 3.4 Производительность БД
**SLI:** Время выполнения SQL-запросов (p95).  
**Метод измерения:** PostgreSQL `pg_stat_statements` или Prometheus-экспортер.

### 3.5 Пропускная способность событий (Kafka)
**SLI:** Задержка доставки событий (от публикации до получения подписчиком).  
**Метод измерения:** Kafka consumer lag (разница между offset производителя и потребителя).

### 3.6 Доступность инфраструктуры
**SLI:** Uptime критических компонентов (Kubernetes pods, БД, Kafka).  
**Метод измерения:** Проверка health-эндпоинтов, мониторинг состояния pods.

## 4. SLO (Service Level Objectives)

### 4.1 Доступность API
- **SLO:** 99.9% успешных запросов в месяц (≈ 8.76 часов простоя в год).
- **Измеряется за:** календарный месяц.

### 4.2 Задержка API
- **SLO для чтения (GET):** 95% запросов < 200 мс.
- **SLO для записи (POST, PUT, DELETE):** 95% запросов < 500 мс.
- **Измеряется за:** 5-минутные окна (rolling).

### 4.3 Частота ошибок
- **SLO:** ошибки (HTTP 5xx) < 1% от всех запросов.
- **Измеряется за:** 5-минутные окна.

### 4.4 Производительность БД
- **SLO:** 95% SQL-запросов выполняются < 100 мс (чтение), < 200 мс (запись).

### 4.5 Задержка событий (Kafka)
- **SLO:** 95% событий доставляются подписчикам за < 1 секунду.

### 4.6 Uptime критических компонентов
- **SLO:** Ключевые pods (backend, frontend) имеют uptime ≥ 99.9% (за исключением планового обслуживания).

## 5. Методы измерения и инструменты

| SLI | Инструмент | Частота сбора | Где смотреть |
|-----|------------|---------------|--------------|
| Доступность API | Prometheus + Blackbox | Каждые 30 сек | Grafana: «API Availability» |
| Задержка API | Prometheus (histogram) | В реальном времени | Grafana: «Latency p95» |
| Ошибки API | Prometheus (counter) | В реальном времени | Grafana: «Error Rate» |
| Производительность БД | PostgreSQL + Prometheus exporter | Каждые 15 сек | Grafana: «DB Query Time» |
| Kafka lag | Kafka exporter + Prometheus | Каждые 30 сек | Grafana: «Kafka Consumer Lag» |
| Uptime pods | Kubernetes API + Prometheus | Каждые 30 сек | Grafana: «Pod Uptime» |

## 6. Алерты и триггеры

| SLO | Порог для алерта | Действие |
|-----|------------------|----------|
| Доступность API | < 99.9% за 5 минут | Отправить алерт в Slack, начать диагностику |
| Задержка API | p95 > 500 мс для чтения / > 1000 мс для записи за 5 минут | Проверить БД, инфраструктуру |
| Ошибки API | > 2% за 5 минут | Изучить логи, эскалировать |
| Задержка БД | p95 > 200 мс за 5 минут | Проверить индексы, нагрузку, блокировки |
| Kafka lag | > 1000 сообщений за 5 минут | Проверить потребителей, увеличить партиции |
| Uptime pod | Pod не доступен более 2 минут | Перезапустить pod, проверить кластер |

Алерты отправляются в **Slack (#alerts)** и по **email** для on-call инженеров.

## 7. Окна для SLO

| SLO | Окно измерения | Допустимый бюджет ошибок |
|-----|----------------|--------------------------|
| Доступность 99.9% | Месяц | 44 минуты простоя в месяц |
| Доступность 99.9% | Квартал | 2.2 часа в квартал |
| Доступность 99.9% | Год | 8.76 часов в год |

Это означает, что мы можем позволить себе 43 минуты недоступности в месяц без нарушения SLO.

---

## 8. План улучшения SLO

- **Текущий уровень:** 99.9% доступности, p95 < 200 мс.
- **Целевой уровень через 6 месяцев:** 99.95% доступности, p95 < 150 мс.
- **Шаги для достижения:**
  - Улучшение мониторинга и алертинга.
  - Оптимизация SQL-запросов, индексов.
  - Внедрение кэширования (Redis) для частых запросов.
  - Масштабирование инфраструктуры (Kubernetes HPA, больше реплик).
