# ⚙️ Операционные документы

**Эта директория содержит документы, описывающие эксплуатацию системы: развертывание, мониторинг, устранение сбоев и обеспечение отказоустойчивости.**  
Здесь собраны ответы на вопросы:
- **Как развернуть систему?** (Deployment Guide)
- **Как действовать при сбоях?** (Runbook)
- **Насколько система надёжна?** (SLO/SLI)
- **Как восстановить систему после катастрофы?** (Disaster Recovery Plan)

Операционные документы — это **инструкции для DevOps, инженеров поддержки и всей команды**, которые обеспечивают стабильную и предсказуемую работу системы в продакшене. Они помогают минимизировать время простоя, быстро реагировать на инциденты и постоянно улучшать надёжность.

## 🧠 Принципы эксплуатации

1. **Автоматизация.** Всё, что можно автоматизировать (сборка, тестирование, деплой, откат), должно быть автоматизировано.
2. **Наблюдаемость (Observability).** Система должна быть прозрачной: логи, метрики, трассировка должны собираться и быть доступными для анализа.
3. **Безопасность.** Доступы, секреты, сетевые политики — строго контролируются и документируются.
4. **Готовность к инцидентам.** Runbook, DRP, тренировки — всё должно быть готово к быстрому реагированию.
5. **Постоянное улучшение.** После каждого инцидента проводится постмортем, результаты которого улучшают процессы и документацию.

## 📄 Описание документов

### 1. Руководство по развертыванию (`deployment.md`)

**Что содержит:**
- **Архитектура развертывания** — схема инфраструктуры (Kubernetes, виртуальные машины, облачные сервисы).
- **Требования к окружению** — необходимые версии ПО, зависимости, переменные окружения.
- **Инструкции по сборке** — как собрать артефакты (бинарные файлы, Docker-образы).
- **Инструкции по развертыванию** — пошаговые команды для dev, staging и production.
- **Инструкции по обновлению** — как выполнить rolling update, blue-green deployment или канареечный релиз.
- **Инструкции по откату** — как быстро вернуться к предыдущей версии.
- **Проверки после деплоя** — как убедиться, что система работает корректно (health checks, smoke tests).

**Когда использовать:**
- При первом развертывании системы.
- При каждом релизе для DevOps и разработчиков.
- При онбординге новых DevOps-инженеров.

**Связь:**  
- Связан с `../project-management/engineering-practices.md` (CI/CD пайплайны).
- Связан с `../architecture/diagrams/c4/` (контейнеры и их расположение).
- Использует `../templates/` для шаблонов конфигураций.

---

### 2. Runbook (`runbook.md`)

**Что содержит:**
- **Типовые инциденты** — падение сервиса, ошибки БД, перегрузка очередей, проблемы с сетью.
- **Симптомы** — как распознать проблему (ошибки в логах, метрики, алерты).
- **Причины** — возможные корневые причины.
- **Действия** — пошаговые инструкции по устранению проблемы (какие команды выполнить, какие логи посмотреть, кого уведомить).
- **Эскалация** — кто отвечает за решение (уровни поддержки, контакты).

**Когда использовать:**
- При возникновении инцидента.
- При обучении новых инженеров поддержки.
- Для регулярных тренировок (Chaos Engineering, Game Days).

**Связь:**  
- Связан с `slo-sli.md` (если SLO нарушены, активируется Runbook).
- Связан с `disaster-recovery.md` (если инцидент перерастает в катастрофу).
- Связан с `../architecture/adr/` (решения, влияющие на отказоустойчивость).

---

### 3. SLO/SLI (`slo-sli.md`)

**Что содержит:**
- **Определения SLI** — какие метрики измеряем (например, время ответа, частота ошибок, доступность).
- **Целевые SLO** — какие значения должны быть гарантированы (например, 99.9% доступность, 95% запросов < 200ms).
- **Методы измерения** — как собираются метрики, какие инструменты (Prometheus, Grafana, ELK).
- **Пороговые значения и алерты** — при каких условиях срабатывают оповещения.
- **Анализ и улучшение** — как пересматриваются SLO и как улучшаются SLI.

**Когда использовать:**
- Для прозрачности надёжности перед бизнесом и стейкхолдерами.
- Для принятия решений о масштабировании и улучшениях.
- Для оценки успешности релизов и изменений.

**Связь:**  
- Связан с `runbook.md` (если SLO нарушены → активируется Runbook).
- Связан с `../project-management/engineering-practices.md` (как тесты и мониторинг влияют на SLI).
- Связан с `../strategic/product-vision.md` (бизнес-цели и ожидания по надёжности).

---

### 4. План восстановления (`disaster-recovery.md`)

**Что содержит:**
- **Сценарии катастроф** — потеря дата-центра, сбой облачного провайдера, повреждение данных, DDoS-атака.
- **RTO (Recovery Time Objective)** — максимальное допустимое время восстановления.
- **RPO (Recovery Point Objective)** — максимально допустимые потери данных.
- **План действий** — пошаговые инструкции по восстановлению инфраструктуры, данных и сервисов.
- **Резервное копирование** — стратегия бэкапов, частота, место хранения, процедуры восстановления.
- **Ответственные** — кто принимает решения, кто выполняет восстановление, кто информирует стейкхолдеров.
- **Тестирование DRP** — как и когда проводятся учения.

**Когда использовать:**
- При серьёзных сбоях, угрожающих работе всей системы.
- Для регулярного тестирования готовности к катастрофам (не реже раза в год).
- Для согласования с бизнесом и стейкхолдерами допустимых потерь и времени простоя.

**Связь:**  
- Связан с `deployment.md` (восстановление инфраструктуры).
- Связан с `../architecture/adr/` (решения по репликации, резервированию).
- Связан с `../strategic/budget-resources.md` (бюджет на резервные мощности).

## 🚀 Как использовать операционные документы

1. **Начни с `deployment.md`** — чтобы развернуть систему в любом окружении.
2. **Ознакомься с `slo-sli.md`** — чтобы понимать, какая надёжность ожидается и измеряется.
3. **Подготовь `runbook.md`** — чтобы быстро реагировать на инциденты.
4. **Разработай `disaster-recovery.md`** — чтобы быть готовым к худшим сценариям.
5. **Регулярно обновляй документы** при изменении архитектуры, инфраструктуры или процессов.

---

## 📌 Важно

- **Операционные документы должны быть практичными.** Инструкции должны быть чёткими, воспроизводимыми и актуальными.
- **Runbook и DRP нужно тестировать** — регулярно проводить учения, чтобы команда знала, что делать в экстренной ситуации.
- **SLO должны быть реалистичными.** Не обещайте 99.99%, если инфраструктура этого не обеспечивает.
- **Документация должна быть доступна** в офлайн-режиме (например, в репозитории) на случай, если внешние системы недоступны.
