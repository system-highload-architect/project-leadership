# 📋 Архитектурные решения (ADR)

**ADR (Architecture Decision Record)** — это документ, который фиксирует важное архитектурное решение, его контекст, альтернативы и обоснование.

ADR — это **основной инструмент для документирования архитектуры**. Он позволяет:
- Сохранять **историю решений** и их обоснование.
- Передавать **контекст** новым членам команды.
- Избегать **повторного обсуждения** уже принятых решений.
- Обосновывать **компромиссы** перед стейкхолдерами.

В этой директории хранятся все ADR проекта. Каждое решение — это отдельный файл с уникальным номером.

## 🧠 Структура ADR

Каждый ADR должен содержать следующие разделы:

```markdown
# ADR-XXX: Название решения

## Статус
[Принято / Предложено / Устарело / Заменено]

## Контекст
Описание проблемы или ситуации, в которой принимается решение.
Что мы знаем? Какие ограничения? Какие требования?

## Решение
Что мы решили сделать? Какое именно решение принято?

## Альтернативы
Какие другие варианты рассматривались? Почему они не подошли?

## Последствия
Какие последствия влечёт это решение:
- Положительные (что улучшится)
- Отрицательные (какие риски или компромиссы)

## Связанные ADR
Ссылки на другие ADR, которые связаны с этим решением.
```

## 🚀 Как создать новый ADR

1. **Скопируй шаблон** `template.md` в новый файл с номером следующего ADR (например, `007-new-decision.md`).
2. **Заполни все разделы**:
   - Дайте чёткое название.
   - Опишите контекст — почему это решение нужно принимать сейчас.
   - Сформулируйте решение — что именно мы делаем.
   - Перечислите альтернативы — что ещё рассматривали и почему не выбрали.
   - Опишите последствия — положительные и отрицательные.
   - Укажите связанные ADR, если есть.
3. **Обнови таблицу** в этом `README.md`, добавив новую строку.
4. **Согласуй решение** с командой и стейкхолдерами (при необходимости).
5. **Зафиксируй статус** — измени `[Предложено]` на `[Принято]` после утверждения.

## 📌 Важные принципы

1. **Каждое значимое архитектурное решение — в ADR.** Если решение влияет на архитектуру, масштабируемость, безопасность, производительность или сложность системы — оно достойно ADR.
2. **ADR пишутся в момент принятия решения.** Не откладывайте на потом — контекст быстро забывается.
3. **ADR — живые документы.** Если решение устарело или заменено — обнови статус и укажи, чем заменено.
4. **ADR должны быть понятны новичкам.** Пиши так, чтобы человек, пришедший в проект через год, мог понять, почему система устроена именно так.
