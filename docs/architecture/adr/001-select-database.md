# ADR-001-select-database.md: Выбор базы данных

**Статус:** Принято  
**Дата:** 2026-07-01  
**Автор:** Tech Lead / Архитектор

---

## Контекст

Для хранения данных в системе RMS (роботы, задачи, заказы, локации) нам необходимо выбрать базу данных. Требования:

- **Транзакционная целостность** — данные должны быть согласованы (ACID).
- **Гибкость** — модель данных будет эволюционировать.
- **Производительность** — система должна выдерживать нагрузку до 1000+ роботов, множество одновременных операций.
- **Надёжность** — резервирование, репликация, бэкапы.
- **Поддержка событийной модели** — возможность интеграции с Kafka / CDC.
- **Опыт команды** — предпочтительны технологии, которые мы уже знаем.

Рассматриваются реляционные и NoSQL-решения.

---

## Решение

Выбираем **PostgreSQL** как основную базу данных.

Обоснование:
- **ACID-транзакции** — критически важны для заказов, задач и статусов роботов.
- **Гибкость JSON** — поддержка типа `JSONB` позволяет хранить полуструктурированные данные, что даёт гибкость на ранних этапах.
- **Надёжность** — репликация (streaming replication), Point-in-Time Recovery (PITR), бэкапы.
- **Производительность** — хорошие индексы (B-tree, GIN для JSON), поддержка партиционирования.
- **CDC (Change Data Capture)** — можно использовать Debezium для публикации изменений в Kafka (интеграция с событийной архитектурой).
- **Опыт команды** — большинство разработчиков знакомы с SQL и PostgreSQL.

---

## Альтернативы

### 1. MySQL
- **Плюсы:** Популярность, привычность.
- **Минусы:** Менее продвинутая поддержка JSON, ограничения в партиционировании, слабее в сложных транзакциях.
- **Почему не выбран:** PostgreSQL даёт больше гибкости и функциональности для нашего сценария.

### 2. MongoDB (NoSQL)
- **Плюсы:** Гибкая схема, горизонтальное масштабирование.
- **Минусы:** Нет полноценных ACID-транзакций (только с версии 4.0, но всё равно сложнее), сложнее с отчетами и связями.
- **Почему не выбран:** Нам важна целостность данных и сложные запросы (JOIN), что лучше делать в реляционной БД.

### 3. CockroachDB / TiDB (NewSQL)
- **Плюсы:** Горизонтальная масштабируемость, совместимость с PostgreSQL.
- **Минусы:** Сложность настройки, меньше готовых инструментов, дороже.
- **Почему не выбран:** На старте нам не требуется горизонтальное масштабирование, а сложность администрирования выше.

## Последствия

### Положительные
- ✅ Мощный и проверенный инструмент с большим сообществом.
- ✅ Поддержка сложных запросов и транзакций.
- ✅ Легко интегрируется с событийной шиной через Debezium.
- ✅ Простота миграций (golang-migrate, etc.).
- ✅ Возможность использования JSONB для гибких данных.

### Отрицательные
- ❌ Вертикальное масштабирование (одна БД) — при росте нагрузки потребуется шардирование или переход на другое решение.
- ❌ Требуется грамотное проектирование индексов и схемы.
- ❌ Операционные расходы (настройка репликации, бэкапов).

### Компромиссы
- Мы проектируем схему с учётом будущего шардирования (используем UUID, избегаем зависимостей между таблицами разных модулей).
- Внедряем кэширование (Redis) для снижения нагрузки на БД.
- На ранних этапах используем один инстанс, но сразу настраиваем репликацию для отказоустойчивости.
