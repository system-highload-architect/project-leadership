# ADR-004: Выбор Clean Architecture как стиля организации кода

**Статус:** Принято  
**Дата:** 2026-07-07  
**Автор:** Tech Lead / Архитектор

---

## Контекст

Мы определили архитектурный стиль системы (модульный монолит, EDA) и стратегию эволюции (SOA → микросервисы). Теперь нам нужно выбрать **стиль организации кода внутри каждого модуля / сервиса**. Ключевые требования:

- Бизнес-логика должна быть **изолирована от внешних зависимостей** (БД, HTTP, gRPC, брокеры).
- Система должна быть **тестируемой** — бизнес-логика должна тестироваться без поднятия инфраструктуры.
- Код должен быть **понятным и поддерживаемым** — новые разработчики должны быстро вникать.
- Модули должны быть **готовы к выделению в микросервисы** — внутренняя структура не должна мешать этому.
- Архитектура должна быть **эволюционной** — мы можем менять детали реализации без переписывания ядра.

---

## Решение

Мы выбираем **Clean Architecture** (с элементами DDD) как основной стиль организации кода для каждого модуля.

Это означает:

1. **Чёткое разделение на слои:**
   - **Domain (Entity)** — бизнес-сущности, правила, Value Objects. Не зависит ни от чего.
   - **Usecase (Application)** — бизнес-логика, оркестрация, команды, запросы. Зависит от Domain и интерфейсов репозиториев.
   - **Controller (Delivery)** — транспортный слой (HTTP, gRPC). Зависит от Usecase.
   - **Infrastructure (Repository)** — реализации внешних зависимостей (БД, брокеры, клиенты). Зависит от интерфейсов Usecase.

2. **Направление зависимостей — внутрь:**
   - Controller → Usecase → Domain
   - Repository → Usecase → Domain
   - Внешние слои зависят от внутренних, но не наоборот.

3. **Интерфейсы определяются внутри Usecase** (а не в Infrastructure), что позволяет легко заменять реализации.

4. **Каждый модуль** имеет свою собственную Clean Architecture внутри (может быть вынесен как отдельный микросервис без изменения структуры).

## Альтернативы

### 1. Layered Architecture (n-tier)
- **Плюсы:** Простота, быстрый старт, привычна многим разработчикам.
- **Минусы:** Бизнес-логика может «расползаться» по слоям, тестируемость ниже (зависит от БД).
- **Почему не выбран:** Мы хотим максимизировать тестируемость и изоляцию бизнес-логики.

### 2. Hexagonal Architecture (Ports & Adapters)
- **Плюсы:** Максимальная изоляция ядра, чёткие порты и адаптеры.
- **Минусы:** Более высокая сложность, больше интерфейсов, требует дисциплины.
- **Почему не выбран:** Clean Architecture даёт достаточную изоляцию, но проще для понимания командой.

### 3. Onion Architecture
- **Плюсы:** Ядро в центре, чёткие кольца зависимостей.
- **Минусы:** Сложность, менее привычна для большинства разработчиков.
- **Почему не выбран:** Слишком сложна для старта, Clean Architecture — золотая середина.

## Последствия

### Положительные
- ✅ **Тестируемость** — Use Cases тестируются с моками репозиториев.
- ✅ **Гибкость** — можно заменить БД, брокер или транспорт без изменения бизнес-логики.
- ✅ **Чёткие границы** — каждый слой имеет свою ответственность.
- ✅ **Готовность к микросервисам** — каждый модуль уже имеет структуру, готовую к выделению.
- ✅ **Понятность** — новые разработчики быстро понимают структуру.

### Отрицательные
- ❌ **Больше кода** — нужно писать интерфейсы, мапперы, DTO.
- ❌ **Дисциплина** — легко нарушить направление зависимостей.
- ❌ **Сложность на старте** — требует времени на проектирование.

### Компромиссы
- Мы используем **Code Generation** (например, `go generate`) для автоматического создания мапперов и DTO.
- Внедряем **архитектурные тесты** с помощью `go-cleanarch`, чтобы проверять направление зависимостей в CI.
- Для простых CRUD-операций допускаем упрощения (например, использование репозитория напрямую в контроллере), но с чёткими границами.

## Пример структуры модуля

```
internal/
├── domain/
│   ├── robot.go               # Сущность Robot
│   └── errors.go              # Доменные ошибки
├── usecase/
│   ├── robot/
│   │   ├── assign_task.go     # Use Case: назначить задачу
│   │   ├── get_robot.go       # Use Case: получить робота
│   │   └── interfaces.go      # Интерфейсы репозиториев и зависимостей
├── delivery/
│   ├── http/
│   │   └── robot_handler.go   # HTTP-хендлер
│   └── grpc/
│       └── robot_server.go    # gRPC-сервер
└── repository/
    ├── postgres/
    │   └── robot_repo.go      # Реализация репозитория через PostgreSQL
    └── interfaces.go          # (опционально, если не определены в usecase)
```
