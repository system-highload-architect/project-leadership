# 🏛️ C4-диаграммы

**Эта директория содержит C4-диаграммы, описывающие архитектуру системы на четырёх уровнях детализации.**  
Модель C4 (Context, Containers, Components, Code) — это подход к визуализации архитектуры, предложенный Саймоном Брауном. Он помогает представить систему на разных уровнях абстракции, от общего контекста до деталей реализации, и делает архитектуру понятной для разных аудиторий.

C4-диаграммы — это **основной инструмент для архитектурной коммуникации**:  
- **Стейкхолдеры и бизнес** видят общую картину (контекст).
- **Архитекторы и Tech Lead** видят контейнеры и компоненты.
- **Разработчики и DevOps** видят детали развёртывания и динамику.

## 🧠 Что такое C4-модель

C4-модель использует **4 уровня абстракции**:

| Уровень | Название | Что показывает | Пример |
|---------|----------|----------------|--------|
| **Уровень 1** | **Context** (Контекст) | Система как «чёрный ящик», её пользователи и внешние системы, с которыми она взаимодействует | «Robot Management System взаимодействует с операторами, администраторами и внешним API погоды» |
| **Уровень 2** | **Containers** (Контейнеры) | Высокоуровневые технологические блоки: веб-приложения, API-сервисы, БД, брокеры сообщений | «Web UI, API Gateway, Order Service, PostgreSQL, Kafka» |
| **Уровень 3** | **Components** (Компоненты) | Внутреннее устройство одного контейнера: основные модули и их ответственность | «В Robot Service: HTTP Handler, Use Cases, Domain, Repository, Postgres Adapter» |
| **Уровень 4** | **Code** (Код) | Детали реализации (классы, функции). Обычно не рисуется, вместо него используют UML-диаграммы | (Опускается, т.к. детали видны в коде и UML) |

## 🧩 Какие диаграммы C4 есть в этой папке

### 1. Контекстная диаграмма (`context.puml`)

**Что показывает:**  
- Нашу систему (Robot Management System) как центральный элемент.
- Пользователей (операторы, администраторы).
- Внешние системы (погодный API, система мониторинга).
- Взаимодействия между ними.

**Когда использовать:**  
- Для знакомства с проектом новых членов команды.
- Для презентаций стейкхолдерам и бизнесу.
- Для определения границ системы и её внешних зависимостей.

**Пример:**  
```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml
Person(user, "Оператор", "Управляет роботами")
System(robot_system, "Robot Management System", "Управляет роботами, задачами и локациями")
System_Ext(api, "External API", "Сторонние сервисы")
Rel(user, robot_system, "Использует", "HTTPS")
Rel(robot_system, api, "Запрашивает данные", "REST/gRPC")
@enduml
```

---

### 2. Диаграмма контейнеров (`containers.puml`)

**Что показывает:**  
- Технологические контейнеры: веб-интерфейс, API-шлюз, сервисы (Order, Robot, Task), БД, брокер сообщений.
- Как они взаимодействуют друг с другом (синхронно, асинхронно).
- Протоколы взаимодействия (HTTPS, gRPC, Kafka).

**Когда использовать:**  
- Для высокоуровневого архитектурного обсуждения.
- Для планирования инфраструктуры и деплоя.
- Для понимания, как данные и команды движутся через систему.

**Пример:**  
```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml
Person(user, "Оператор")
System_Boundary(robot_system, "Robot Management System") {
    Container(web_ui, "Web UI", "React")
    Container(api_gateway, "API Gateway", "Go + Gin")
    Container(order_service, "Order Service", "Go")
    Container(robot_service, "Robot Service", "Go")
    ContainerDb(postgres, "PostgreSQL", "БД")
    Container(kafka, "Kafka", "Брокер событий")
}
Rel(user, web_ui, "Использует", "HTTPS")
Rel(web_ui, api_gateway, "Вызывает", "HTTPS")
Rel(api_gateway, order_service, "Маршрутизирует", "gRPC")
Rel(order_service, robot_service, "Вызывает", "gRPC")
Rel(order_service, postgres, "Читает/пишет", "SQL")
Rel(order_service, kafka, "Публикует события", "Kafka")
@enduml
```

---

### 3. Диаграмма компонентов (`components.puml`)

**Что показывает:**  
- Внутреннее устройство одного контейнера (например, Robot Service).
- Компоненты: HTTP-хендлеры, use cases, домен, репозитории, адаптеры.
- Зависимости между ними (кто кого вызывает, кто кого реализует).

**Когда использовать:**  
- При детальном проектировании микросервиса или модуля.
- При код-ревью и рефакторинге.
- Для понимания, как реализован конкретный контейнер.

**Пример:**  
```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Component.puml
Container(robot_service, "Robot Service", "Go")
Component(handler, "HTTP Handler", "Gin")
Component(usecase, "Use Cases", "Go")
Component(domain, "Domain", "Go")
Component(repo, "Repository", "Go")
Component(postgres_adapter, "Postgres Adapter", "Go")
Rel(handler, usecase, "Вызывает")
Rel(usecase, domain, "Использует")
Rel(usecase, repo, "Вызывает")
Rel(postgres_adapter, repo, "Реализует")
@enduml
```

## ⚖️ C4 vs UML

| Критерий | C4 | UML |
|----------|----|-----|
| **Уровень** | От контекста до компонентов | Классы, последовательности, состояния |
| **Аудитория** | Все: от бизнеса до разработчиков | Разработчики и архитекторы |
| **Фокус** | Архитектура системы в целом | Детали кода и взаимодействий |
| **Когда использовать** | Для высокоуровневого проектирования и коммуникации | Для детального моделирования и реализации |

C4-диаграммы и UML-диаграммы **дополняют друг друга**. C4 даёт общую картину, UML — детали.

---

## 🚀 Как использовать C4-диаграммы

1. **Начни с контекстной диаграммы** — это входная точка для всех.
2. **Перейди к диаграмме контейнеров** — чтобы понять технологический ландшафт.
3. **Для каждого ключевого контейнера создай диаграмму компонентов** — чтобы показать внутреннее устройство.
4. **Для низкоуровневых деталей** используй UML-диаграммы (классов, последовательности, состояний).

---

## 📌 Важно

- **Все C4-диаграммы должны быть синхронизированы с реальной архитектурой.** Если меняется система, обновляй диаграммы.
- **Используй стандартные нотации C4** (люди, системы, контейнеры, компоненты). Это делает диаграммы понятными для всех.
- **Не перегружай диаграммы** — каждая должна показывать ровно столько деталей, сколько нужно для её уровня.
- **Сопровождай диаграммы текстовыми описаниями** в README или ADR — поясняй, что показано и почему.
