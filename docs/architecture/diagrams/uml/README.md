# 📐 UML-диаграммы

**Эта директория содержит UML-диаграммы, описывающие структуру и поведение системы на уровне кода и взаимодействий.**  
UML (Unified Modeling Language) — это стандартный язык визуального моделирования, который помогает разработчикам и архитекторам понимать, проектировать и документировать программные системы.

В этой папке собраны три основных типа UML-диаграмм, которые наиболее полезны для проектирования и разработки:
- **Диаграммы классов** — статическая структура системы (сущности, атрибуты, методы, связи).
- **Диаграммы последовательности** — динамика взаимодействия объектов во времени.
- **Диаграммы состояний** — жизненные циклы сущностей и переходы между состояниями.

## 🧠 Принципы построения UML-диаграмм

1. **Диаграмма классов = отражение доменной модели.** Она должна соответствовать структуре кода в `internal/domain/` и `internal/usecase/`. Изменения в коде → изменения в диаграмме.
2. **Диаграмма последовательности = ключевые сценарии.** Описывает наиболее важные бизнес-процессы (например, создание заказа, назначение задачи). Не нужно описывать все возможные сценарии — только критические.
3. **Диаграмма состояний = жизненный цикл.** Для каждой сущности с изменяемым статусом (робот, задача, заказ) должна быть диаграмма состояний.
4. **Единообразие.** Все UML-диаграммы используют единые нотации и стили (PlantUML).
5. **Синхронизация с ADR.** Если архитектурное решение меняет структуру классов или взаимодействия — обнови диаграммы.

## 🧩 Описание диаграмм

### 1. Диаграмма классов (`class.puml`)

Показывает статическую структуру системы: классы, их атрибуты, методы и связи между ними.

**Основные элементы:**
- **Классы** — сущности (например, `Robot`, `Task`, `Order`, `Location`).
- **Атрибуты** — поля класса (например, `status`, `name`, `priority`).
- **Методы** — функции класса (например, `assignTask()`, `changeStatus()`).
- **Связи** — ассоциации, агрегации, композиции, наследования.

**Пример фрагмента:**
```plantuml
class Robot {
  - id: UUID
  - name: string
  - status: RobotStatus
  + assignTask(task: Task): void
  + changeStatus(newStatus: RobotStatus): void
}
Robot "1" -- "0..*" Task : assigned
```

**Когда использовать:** При проектировании доменной модели, рефакторинге, код-ревью, знакомстве новых разработчиков с системой.

---

### 2. Диаграмма последовательности (`sequence.puml`)

Показывает, как объекты взаимодействуют во времени при выполнении конкретного сценария.

**Основные элементы:**
- **Акторы** — внешние сущности (пользователь, система).
- **Объекты/компоненты** — участники взаимодействия (API Gateway, сервисы, БД).
- **Сообщения** — вызовы методов или запросы.
- **Временные линии** — жизнь каждого участника.

**Пример фрагмента:**
```plantuml
actor User
participant "API Gateway" as Gateway
participant "Order Service" as Order
participant "Robot Service" as Robot

User -> Gateway: POST /orders
Gateway -> Order: CreateOrder
Order -> Robot: AssignRobot
```

**Когда использовать:** Для описания ключевых бизнес-процессов, API-вызовов, интеграций, отладки распределённых транзакций.

---

### 3. Диаграмма состояний (`state.puml`)

Показывает жизненный цикл сущности: возможные состояния, переходы между ними и события, вызывающие переходы.

**Основные элементы:**
- **Состояния** — этапы жизни (например, `CREATED`, `IN_PROGRESS`, `COMPLETED`).
- **Переходы** — стрелки между состояниями.
- **События** — триггеры переходов (например, `задача назначена`, `все задачи выполнены`).

**Пример фрагмента:**
```plantuml
[*] --> CREATED : Создан
CREATED --> IN_PROGRESS : Назначена задача
IN_PROGRESS --> COMPLETED : Все задачи выполнены
IN_PROGRESS --> CANCELLED : Отменён
```

**Когда использовать:** Для моделирования статусов сущностей с изменяемым состоянием (заказы, задачи, роботы). Особенно полезно для реализации бизнес-логики.

## 🚀 Как использовать UML-диаграммы

### 1. Разработка и проектирование
- **Начни с диаграммы классов** — это основа всей системы.
- **Добавь диаграммы последовательности** для ключевых сценариев (например, создание заказа, назначение задачи, завершение работы).
- **Опиши диаграммы состояний** для сущностей с изменяемым статусом.

### 2. Онбординг новых разработчиков
- Новый член команды начинает с диаграммы классов — видит общую структуру.
- Затем переходит к диаграммам последовательности — понимает потоки данных.
- Диаграммы состояний помогают понять бизнес-логику и статусы.

### 3. Код-ревью и рефакторинг
- Если код меняется — проверь, нужно ли обновить диаграммы.
- Если диаграммы устарели — это сигнал, что код и архитектура разошлись.

### 4. Документирование архитектуры
- UML-диаграммы — это часть архитектурной документации. Они дополняют ADR и текстовые описания.

---

## 📌 Важно

- **Диаграммы классов должны соответствовать коду.** Если код изменился — обнови диаграмму.
- **Диаграммы последовательности описывают только ключевые сценарии.** Не нужно детализировать каждый возможный путь.
- **Диаграммы состояний обязательны** для сущностей с бизнес-статусами.
- **Все изменения в UML должны быть синхронизированы с ADR.** Если принято новое решение, влияющее на структуру или поведение — обнови диаграммы и запиши ADR.
