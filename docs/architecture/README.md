# 🏗️ Архитектурные документы

**Эта директория содержит все технические артефакты, описывающие архитектуру системы.**  
Здесь собраны решения, диаграммы, контракты и модели данных, которые определяют, **как устроена система** и **почему она устроена именно так**.

Документы в этой папке — это **основной источник истины** для архитекторов, разработчиков, DevOps-инженеров и всех, кто участвует в создании и развитии системы.

---

## 🧠 Принципы ведения архитектурной документации

1. **Каждое важное решение — в ADR.** Если решение влияет на архитектуру, оно должно быть задокументировано в ADR с указанием контекста, альтернатив и обоснования.
2. **Диаграммы — не картинки, а код.** Все диаграммы хранятся в текстовом формате (PlantUML) и генерируются автоматически в SVG/PNG.
3. **API-контракты — единый источник.** Все спецификации API хранятся в этой папке и используются для генерации клиентов и документации.
4. **Модель данных — живой документ.** Схема БД и ER-диаграммы обновляются при каждом изменении структуры данных.
5. **Документация — часть кода.** Все архитектурные артефакты хранятся в репозитории рядом с кодом и версионируются вместе с ним.

## 🚀 Как использовать архитектурные документы

1. **Начни с ADR** — изучи уже принятые решения, чтобы понять контекст и обоснования.
2. **Посмотри диаграммы** — чтобы визуально понять структуру системы и взаимодействия.
3. **Изучи API-контракты** — чтобы знать, как взаимодействовать с системой извне.
4. **Ознакомься с моделью данных** — чтобы понимать структуру хранения и связи между сущностями.
5. **При принятии нового решения** — создай новый ADR по шаблону `template.md` и обнови соответствующие диаграммы и схемы.

---

## 📌 Важно

- **ADR обязательны для всех значимых изменений.** Даже если решение кажется очевидным — задокументируй его.
- **Диаграммы должны быть синхронизированы с кодом.** Если меняется архитектура — меняются диаграммы.
- **API-контракты — это договор.** Их изменение должно быть согласовано со всеми потребителями API.
