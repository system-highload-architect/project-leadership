# 🗄️ Модель данных

**Эта директория содержит описание структуры данных, которые используются в системе.**  
Здесь собраны концептуальная, логическая и физическая модели, а также SQL-схемы для создания базы данных.

Модель данных — это **фундамент системы**. Она определяет, как хранятся сущности, как они связаны друг с другом и какие ограничения накладываются на данные.

## 🧠 Основные сущности

Система оперирует четырьмя ключевыми сущностями:

### 1. Робот (`robots`)

Робот — это исполнитель, который выполняет задачи.

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `id` | UUID | Уникальный идентификатор (PK) |
| `name` | VARCHAR(100) | Название робота |
| `type` | VARCHAR(50) | Тип робота (доставка, сборка, инспекция) |
| `status` | VARCHAR(20) | Текущий статус (idle, busy, maintenance, offline) |
| `location_id` | UUID | Ссылка на локацию (FK) |
| `created_at` | TIMESTAMP | Дата создания |
| `updated_at` | TIMESTAMP | Дата последнего обновления |

**Связи:**
- Может находиться в одной локации (`location_id` → `locations.id`).
- Может выполнять множество задач (`tasks.robot_id`).

---

### 2. Задача (`tasks`)

Задача — это единица работы, которая должна быть выполнена роботом.

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `id` | UUID | Уникальный идентификатор (PK) |
| `description` | TEXT | Описание задачи |
| `status` | VARCHAR(20) | Статус (pending, in_progress, completed, failed) |
| `priority` | INTEGER | Приоритет (чем выше, тем важнее) |
| `order_id` | UUID | Ссылка на заказ (FK) |
| `robot_id` | UUID | Ссылка на робота, выполняющего задачу (FK) |
| `location_id` | UUID | Ссылка на локацию выполнения (FK) |
| `created_at` | TIMESTAMP | Дата создания |
| `updated_at` | TIMESTAMP | Дата последнего обновления |

**Связи:**
- Принадлежит одному заказу (`order_id` → `orders.id`).
- Может быть назначена одному роботу (`robot_id` → `robots.id`).
- Выполняется в одной локации (`location_id` → `locations.id`).

---

### 3. Заказ (`orders`)

Заказ — это контейнер для одной или нескольких задач, объединённых общей целью.

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `id` | UUID | Уникальный идентификатор (PK) |
| `status` | VARCHAR(20) | Статус заказа (created, in_progress, completed, cancelled) |
| `customer_id` | UUID | Внешний идентификатор заказчика |
| `created_at` | TIMESTAMP | Дата создания |
| `updated_at` | TIMESTAMP | Дата последнего обновления |

**Связи:**
- Содержит множество задач (`tasks.order_id`).

---

### 4. Локация (`locations`)

Локация — это физическое или логическое место, где роботы работают и где выполняются задачи.

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `id` | UUID | Уникальный идентификатор (PK) |
| `name` | VARCHAR(100) | Название локации |
| `address` | TEXT | Адрес или описание местоположения |
| `capacity` | INTEGER | Максимальное количество роботов, которые могут одновременно находиться в локации |
| `created_at` | TIMESTAMP | Дата создания |
| `updated_at` | TIMESTAMP | Дата последнего обновления |

**Связи:**
- Содержит множество роботов (`robots.location_id`).
- Содержит множество задач (`tasks.location_id`).

## 🧩 Связи между сущностями

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  Order   │ 1   0..*│   Task   │ 0..*   1│  Robot   │
│          │─────────│          │─────────│          │
└──────────┘         └────┬─────┘         └──────────┘
                          │
                          │ 0..*
                          │
                     ┌────┴─────┐
                     │ Location │
                     │          │
                     └──────────┘
```

- **Order → Task**: один заказ может содержать много задач (1 : 0..*).
- **Task → Robot**: одна задача может быть назначена одному роботу, но робот может выполнять много задач (0..* : 1).
- **Task → Location**: одна задача выполняется в одной локации, в локации может выполняться много задач (0..* : 1).
- **Robot → Location**: робот находится в одной локации, в локации может быть много роботов (0..* : 1).

## 🛠️ Принципы проектирования схемы БД

1. **Нормализация до 3NF.** Избегаем избыточности, но допускаем денормализацию для производительности (например, в read-моделях).
2. **Индексы.** Создаём индексы на поля, используемые в `WHERE`, `JOIN`, `ORDER BY`:
   - `orders.customer_id`
   - `orders.status`
   - `tasks.order_id`
   - `tasks.robot_id`
   - `tasks.location_id`
   - `robots.location_id`
3. **Внешние ключи.** Используем для обеспечения ссылочной целостности с каскадными действиями:
   - `orders` → `tasks`: `ON DELETE CASCADE` (при удалении заказа удаляются задачи).
   - `robots` → `tasks`: `ON DELETE SET NULL` (при удалении робота задачи остаются без исполнителя).
4. **Версионирование.** Поля `created_at` и `updated_at` обязательны для всех таблиц.
5. **Использование UUID.** Все первичные ключи — UUID (генерируются на стороне приложения для избежания коллизий в распределённых системах).
6. **Enum-типы.** Для статусов используются ограничения `CHECK` или отдельные таблицы-справочники.

## 🚀 Как использовать модель данных

1. **Для разработчиков:** Изучи `schema.sql`, чтобы понять структуру таблиц и ограничения. Используй ER-диаграммы для визуализации связей.
2. **Для DBA:** Используй `schema.sql` для развёртывания БД в окружениях (dev, staging, production). Применяй миграции через инструменты (например, `golang-migrate`).
3. **Для аналитиков:** Концептуальная и логическая модели помогут понять бизнес-сущности и их взаимосвязи.
4. **При изменении схемы:** Обнови `schema.sql`, а также соответствующие ER-диаграммы в `diagrams/er/`. Создай новый ADR, если изменение существенное.

---

## 📌 Важно

- **Схема БД — живой документ.** При каждом изменении структуры данных обновляй `schema.sql` и диаграммы.
- **Миграции должны быть обратимыми.** Все изменения схемы должны иметь `up` и `down` скрипты.
- **Тестируй изменения.** Перед применением в продакшене проверяй миграции на staging-окружении.
