# Онбординг: RMS (Robot Management System)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Утверждено

---

## 1. Введение

Данный документ описывает **процесс введения нового сотрудника** в команду RMS. Онбординг — это критически важный этап, который определяет, насколько быстро и эффективно новичок включится в работу, поймёт архитектуру, процессы и культуру команды.

Цель онбординга:
- Сократить время адаптации (до полноценной работы — 2–4 недели).
- Дать новичку всю необходимую информацию и доступы.
- Познакомить с командой, процессами и проектом.
- Снизить стресс и неуверенность на старте.

---

## 2. Роли в онбординге

| Роль | Ответственность |
|------|-----------------|
| **Ментор (назначается Tech Lead)** | Основной наставник, проводит встречи, отвечает на вопросы, помогает с задачами |
| **Tech Lead** | Определяет ментора, утверждает план, контролирует прогресс |
| **HR / People Manager** | Организует доступы, оборудование, знакомство с офисом/удалёнкой |
| **Команда** | Приветствует, помогает в рабочих вопросах, приглашает на встречи |

---

## 3. Этапы онбординга

### 3.1 До первого рабочего дня
| Действие | Ответственный | Срок |
|----------|---------------|------|
| Отправить welcome-письмо с информацией о проекте, ссылками на документы | HR / Tech Lead | За 2–3 дня до старта |
| Оформить доступы (GitHub, Jira, Slack, облако, БД, Kafka, etc.) | DevOps / HR | За 2 дня до старта |
| Подготовить оборудование (ноутбук, мониторы, если офис) | HR | За 1 день до старта |
| Назначить ментора | Tech Lead | За 1 день до старта |

---

### 3.2 Первая неделя (погружение)

**День 1: Знакомство**
- Встреча с Tech Lead и HR (обсуждение проекта, роли, ожиданий).
- Вводное знакомство с командой (представление в Slack / на Daily).
- Установка и настройка рабочего окружения (IDE, Docker, kubectl, etc.).
- Предоставление доступа к репозиторию, документации, задачам.
- Знакомство с расписанием встреч (Daily, Planning, Retro, etc.).

**День 2–3: Изучение документации**
- Прочитать [README.md](../../../README.md) проекта.
- Изучить [стратегические документы](../../strategic/product-vision.md, roadmap.md, architecture-strategy.md, budget-resources.md).
- Просмотреть [архитектурные документы](../../architecture/README.md), особенно ADR и диаграммы (C4, ER, UML).
- Ознакомиться с [инженерными практиками](../project-management/engineering-practices.md) и [регламентами встреч](meetings-guidelines.md).

**День 4–5: Работа с кодом**
- Склонировать репозиторий, собрать проект локально (`make build`).
- Запустить локальное окружение (Docker Compose) и проверить, что всё работает.
- Выполнить первую маленькую задачу (например, исправить баг или добавить небольшой эндпоинт) под руководством ментора.
- Пройти код-ревью и исправить замечания (ментор помогает).

---

### 3.3 Вторая неделя (включение в процессы)

- Активное участие в Daily Standup (отчитываться о прогрессе).
- Выполнение одной средней задачи (например, добавить новый use case) с минимальной поддержкой ментора.
- Самостоятельное создание PR и участие в ревью (как автор и как ревьюер).
- Знакомство с CI/CD (просмотр пайплайнов, логов деплоя).
- Участие в Planning и Retro (наблюдение, затем активное участие).

---

### 3.4 Третий-четвёртый этап (недели 3–4)

- Полноценное включение в спринт: выполнение задач из бэклога, участие в проектировании.
- Презентация своих изменений на демо (если есть что показать).
- Обсуждение с ментором первых итогов: что получается, какие сложности.
- Корректировка индивидуального плана развития (если нужно).
- Назначение первой задачи без ментора (полная самостоятельность).

## 4. Чеклист онбординга (для ментора и новичка)

### Доступы
- [ ] GitHub (репозиторий, actions, packages)
- [ ] Slack / Telegram (командный чат, каналы)
- [ ] Jira / Trello (задачи, бэклог)
- [ ] Облачный провайдер (AWS/GCP, консоль или kubectl)
- [ ] База данных (подключение к dev/staging)
- [ ] Kafka (топики, консоль)
- [ ] Мониторинг (Grafana, Prometheus)
- [ ] Логи (ELK / Loki)

### Документация (прочитано)
- [ ] README.md
- [ ] product-vision.md, roadmap.md, architecture-strategy.md
- [ ] ADR (первые 3–4)
- [ ] Диаграммы C4, ER, UML
- [ ] API-контракты (OpenAPI, Protobuf)
- [ ] Инженерные практики
- [ ] Регламенты встреч

### Практические шаги
- [ ] Установить Go, Docker, kubectl, IDE (VS Code / GoLand)
- [ ] Склонировать репозиторий
- [ ] Собрать проект локально
- [ ] Запустить docker-compose (БД, Kafka, Redis)
- [ ] Запустить сервис локально, выполнить health-check
- [ ] Пройти интеграционные тесты (локально)
- [ ] Создать первый PR (исправление README / мелкий баг)
- [ ] Получить аппрув и смержить PR
- [ ] Выполнить первую задачу из бэклога (с ментором)
- [ ] Выполнить первую самостоятельную задачу

## 6. Обратная связь и адаптация

- **Раз в неделю** — встреча с ментором (15–20 мин) для обсуждения прогресса.
- **Через 2 недели** — встреча с Tech Lead для оценки адаптации.
- **Через 1 месяц** — формальная оценка онбординга, составление плана развития.

---

## 7. Критерии успешного онбординга

- Новичок выполняет задачи из бэклога без постоянной помощи ментора.
- Участвует в Daily, Planning, Retro, Code Review.
- Понимает архитектуру и может объяснить её другому.
- Написал и смержил минимум 3 PR.
- Отзывы от команды положительные.
