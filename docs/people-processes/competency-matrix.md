# Матрица компетенций: RMS (Robot Management System)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

Матрица компетенций описывает **навыки и уровни владения**, необходимые для каждой роли в команде RMS. Она используется для:
- Оценки текущего уровня команды.
- Планирования развития сотрудников.
- Найма новых специалистов.
- Определения гэпов (пробелов) в навыках.

Матрица включает **технические**, **архитектурные**, **продуктовые** и **управленческие** компетенции.

---

## 2. Роли в команде

| Роль | Описание | Ответственность |
|------|----------|-----------------|
| **Tech Lead / Архитектор** | Руководит технической стратегией, принимает архитектурные решения, развивает команду | Архитектура, ADR, код-ревью, наставничество |
| **Backend-разработчик** | Разрабатывает API, бизнес-логику, интеграции | Реализация use cases, работа с БД, событиями |
| **Frontend-разработчик** | Разрабатывает Web UI для операторов и администраторов | Интерфейсы, интеграция с API, UX |
| **DevOps-инженер** | Управляет инфраструктурой, CI/CD, мониторингом | Kubernetes, Docker, облачные провайдеры, автоматизация |
| **QA-инженер** | Обеспечивает качество продукта, автоматизирует тестирование | Тест-дизайн, интеграционные тесты, E2E-тесты, нагрузочное тестирование |
| **Product Manager** | Управляет требованиями, коммуникацией со стейкхолдерами | Бэклог, roadmap, встречи, приоритеты |
| **Аналитик** | Собирает требования, моделирует данные, документирует | Требования, модель данных, документация |

## 3. Компетенции и уровни

### 3.1 Технические компетенции (бэкенд)

| Компетенция | Junior | Middle | Senior | Lead |
|-------------|--------|--------|--------|------|
| **Go** | Базовый синтаксис, стандартная библиотека | Конкурентность, интерфейсы, пакеты | Профилирование, оптимизация, generics | Архитектура на Go, code generation |
| **REST API** | Базовые эндпоинты, HTTP-методы | OpenAPI, валидация, middleware | Проектирование API, версионирование | Стратегия API, документация |
| **gRPC** | Базовое понимание | Реализация сервисов, интерцепторы | Производительность, стриминг | Стратегия gRPC vs REST |
| **SQL / PostgreSQL** | Базовые запросы | Индексы, JOIN, транзакции | Оптимизация, партиционирование, репликация | Архитектура БД, миграции |
| **Kafka** | Базовое понимание | Публикация/подписка, топики | Проектирование событий, партиции | Стратегия EDA, гарантии доставки |
| **Docker** | Базовое использование | Сборка образов, docker-compose | Оптимизация образов, безопасность | Стратегия контейнеризации |
| **Kubernetes** | Базовое понимание | Развертывание Pod'ов, Service'ов | Helm, операторы, сетевые политики | Архитектура кластера, безопасность |
| **CI/CD** | Базовое понимание | GitHub Actions, пайплайны | Канареечные релизы, rollback | Стратегия доставки |

### 3.2 Архитектурные компетенции

| Компетенция | Junior | Middle | Senior | Lead |
|-------------|--------|--------|--------|------|
| **Clean Architecture** | Понимание слоёв | Реализация внутри модуля | Проектирование модулей | Стратегия для всей системы |
| **Event-Driven (EDA)** | Понимание событий | Работа с брокерами | Проектирование событийных потоков | Стратегия EDA, гарантии |
| **CQRS / Event Sourcing** | Понимание концепций | Реализация простых сценариев | Проектирование read/write моделей | Стратегия разделения |
| **Микросервисы** | Понимание концепций | Разработка микросервиса | Проектирование границ сервисов | Стратегия перехода от монолита |
| **DDD** | Понимание Bounded Context | Моделирование в одном модуле | Проектирование контекстов | Стратегия DDD на уровне системы |

### 3.3 Продуктовые и управленческие компетенции

| Компетенция | Junior | Middle | Senior | Lead |
|-------------|--------|--------|--------|------|
| **Управление требованиями** | Понимание задач | Участие в уточнении требований | Составление технического задания | Стратегия продукта |
| **Коммуникация** | Участие в обсуждениях | Активное участие в Daily | Ведение встреч, переговоры | Коммуникация со стейкхолдерами |
| **Наставничество** | Обучение у старших | Помощь новичкам | Менторство джунов | Развитие команды |
| **Документирование** | Создание простых документов | Ведение ADR, README | Проектирование документации | Стратегия документации |
| **Управление рисками** | Понимание рисков | Идентификация рисков | Управление техническими рисками | Стратегия риск-менеджмента |

## 4. Уровни владения

| Уровень | Описание | Пример |
|---------|----------|--------|
| **1 — Базовый** | Знание основ, может выполнять простые задачи под руководством | Может написать простой REST-эндпоинт |
| **2 — Средний** | Самостоятельно выполняет задачи, знает best practices | Может спроектировать API, написать интеграционные тесты |
| **3 — Продвинутый** | Глубокое понимание, может обучать других, решает сложные задачи | Может оптимизировать производительность, проектировать архитектуру модуля |
| **4 — Эксперт** | Стратегическое мышление, влияет на всю систему, предлагает решения | Принимает архитектурные решения на уровне системы |

## 6. Как использовать матрицу

1. **Оценка текущего уровня** — каждый член команды оценивает себя и обсуждает с Tech Lead.
2. **Выявление гэпов** — сравнение текущего уровня с целевым.
3. **Планирование развития** — составление [индивидуального плана развития](individual-development-plan.md) для закрытия гэпов.
4. **Найм** — определение необходимых навыков для новых сотрудников.
5. **Повышение** — прозрачные критерии для перехода на следующий уровень.
