# Дорожная карта: Robot Management System (RMS)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

Дорожная карта определяет **этапы развития** продукта, ключевые вехи и сроки. Она служит инструментом синхронизации команды, менеджеров и стейкхолдеров относительно того, **что и когда** будет реализовано.

Карта построена по принципу **MVP-подхода** — сначала базовая работоспособность, затем наращивание функциональности и масштабируемости. Все этапы соответствуют стратегическим целям, описанным в [product-vision.md](product-vision.md).

---

## 2. Этапы развития

### 🚀 Этап 1: Фундамент (Q3 2026) – 3 месяца
**Цель:** Создать работоспособное ядро системы, достаточное для пилотных запусков.

| Веха | Описание | Срок |
|------|----------|------|
| **Архитектура** | Утверждены ADR, диаграммы C4/ER/UML, модель данных | Неделя 1–2 |
| **Настройка инфраструктуры** | Репозиторий, CI/CD, база данных, брокер (Kafka) | Неделя 2–3 |
| **Базовые сущности** | Реализованы модели Robot, Task, Order, Location | Неделя 3–4 |
| **REST API (MVP)** | CRUD для роботов, задач, заказов, локаций | Неделя 4–6 |
| **Базовое событийное ядро** | Публикация событий при изменениях (Kafka) | Неделя 6–7 |
| **Интерфейс оператора** | Web UI для управления роботами и задачами (базовый) | Неделя 7–9 |
| **Документация** | API-контракты (OpenAPI), первые ADR, README | Неделя 9–10 |
| **Пилот с внутренним пользователем** | Тестирование на реальных сценариях, сбор обратной связи | Неделя 10–12 |

**Результат этапа:** Рабочий прототип, который можно показать первым клиентам.

---

### 🔧 Этап 2: Надёжность и масштабируемость (Q4 2026) – 3 месяца
**Цель:** Улучшить производительность, надёжность и добавить ключевые паттерны.

| Веха | Описание | Срок |
|------|----------|------|
| **Circuit Breaker & Retry** | Защита от сбоев внешних сервисов | Неделя 1–2 |
| **Saga Orchestration** | Распределённые транзакции для сложных заказов | Неделя 2–4 |
| **Мониторинг и логирование** | Настройка Prometheus, Grafana, ELK/Loki | Неделя 4–5 |
| **SLO/SLI** | Определены метрики, настроены алерты | Неделя 5–6 |
| **Оптимизация БД** | Индексы, кэширование (Redis), подготовка к шардированию | Неделя 6–8 |
| **API Gateway** | Единая точка входа, авторизация, rate limiting | Неделя 8–10 |
| **Тренинг команды** | Онбординг, технические сессии, документация | Неделя 10–12 |

**Результат этапа:** Система готова к эксплуатации с гарантированной надёжностью 99.9%.

---

### 🌍 Этап 3: Интеграции и масштабирование (Q1 2027) – 3 месяца
**Цель:** Подключение реальных роботов, расширение API, подготовка к микросервисам.

| Веха | Описание | Срок |
|------|----------|------|
| **Адаптеры для конкретных роботов** | Интеграция с 3 типами роботов (разные API) | Неделя 1–3 |
| **Расширение событийной модели** | Новые события, проекции для аналитики | Неделя 3–5 |
| **CQRS (Read/Write разделение)** | Оптимизация чтения для дашбордов | Неделя 5–7 |
| **Гибридный подход** | Вынос отдельных функций в Serverless (AWS Lambda) | Неделя 7–9 |
| **Масштабирование** | Подготовка к переходу на микросервисы (разделение модулей) | Неделя 9–11 |
| **Пилот с внешними клиентами** | Подключение 5 клиентов в боевом режиме | Неделя 11–12 |

**Результат этапа:** Система масштабируется до 1000+ роботов, готова к промышленной эксплуатации.

---

### 📈 Этап 4: Продуктовая зрелость (Q2 2027 – и далее)
**Цель:** Дальнейшее развитие, аналитика, новые сценарии.

| Веха | Описание | Срок |
|------|----------|------|
| **Умная диспетчеризация** | Алгоритмы автоматического назначения задач | Постоянно |
| **Аналитические дашборды** | Визуализация KPI, эффективности роботов | Постоянно |
| **Интеграция с ERP/WMS** | Стандартные коннекторы | Постоянно |
| **Платформа расширений** | Плагины для пользовательских сценариев | Постоянно |

| Веха | Дата | Критерий успеха |

## 4. Ресурсы

Для выполнения этапов требуется:

| Ресурс | Q3 2026 | Q4 2026 | Q1 2027 | Q2 2027+ |
|--------|---------|---------|---------|----------|
| **Разработчики** | 2 (Backend) + 1 (Frontend) | 3 (Backend) + 1 (Frontend) | 4 (Backend) + 1 (Frontend) + 1 (DevOps) | 5 + 1 + 1 |
| **DevOps** | 0.5 | 0.5 | 1 | 1 |
| **Бюджет инфраструктуры** | $500/мес | $1000/мес | $2000/мес | $3000/мес |
| **Время на документацию** | 20% | 15% | 10% | 10% |

## 5. Риски и смягчения

| Риск | Этап | Меры |
|------|------|------|
| Задержка интеграции с реальными роботами | Этап 3 | Ранний доступ к SDK/API от производителей |
| Рост нагрузки выше ожидаемого | Этап 2 | Готовые оптимизации (кэши, шардирование) |
| Изменение требований | Все | Гибкий процесс, частые итерации |
| Потеря ключевых сотрудников | Все | Документация, code reviews, онбординг |
