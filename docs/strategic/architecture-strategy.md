# Архитектурная стратегия: Robot Management System (RMS)

**Версия:** 1.0  
**Дата:** 2026-07-22  
**Статус:** Черновик

---

## 1. Введение

Архитектурная стратегия определяет **ключевые технологические и архитектурные принципы**, на которых строится система. Она служит ориентиром для всех технических решений, обеспечивая согласованность, масштабируемость и эволюционируемость системы.

Данный документ описывает **выбранные подходы, компромиссы и обоснования** на высоком уровне. Детали каждого конкретного решения фиксируются в отдельных ADR (Архитектурных записях).

---

## 2. Архитектурные принципы

| № | Принцип | Описание |
|---|---------|----------|
| 1 | **Модульность** | Система строится из слабо связанных модулей с чёткими границами (модульный монолит как старт). |
| 2 | **Событийность** | Все изменения состояния публикуются как события (Event-Driven). Это обеспечивает слабую связанность и аудит. |
| 3 | **Чистота кода** | Внутри каждого модуля/сервиса используется Clean Architecture (или Hexagonal) для изоляции бизнес-логики от инфраструктуры. |
| 4 | **Наблюдаемость** | Логи, метрики, трассировка — обязательны с первого дня. |
| 5 | **Эволюционность** | Архитектура проектируется с учётом будущего роста: от монолита к микросервисам без переписывания. |
| 6 | **API-first** | Контракты (OpenAPI, Protobuf) проектируются до реализации. |
| 7 | **Безопасность по умолчанию** | Аутентификация, авторизация, шифрование — встроены на всех уровнях. |
| 8 | **Автоматизация** | CI/CD, тестирование, деплой — максимально автоматизированы. |

## 3. Технологический стек

### Бэкенд
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Язык** | Go | Простота, производительность, отличная поддержка конкурентности, богатая стандартная библиотека |
| **Веб-фреймворк** | Gin (или net/http) | Лёгкий, быстрый, минималистичный |
| **gRPC** | gRPC-Go | Высокая производительность, строгие контракты, поддержка стриминга |
| **ORM / Query** | pgx (нативный драйвер) + sqlx | Полный контроль над SQL, производительность |
| **Миграции** | golang-migrate | Простота, версионирование схемы |

### Хранилища данных
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Основная БД** | PostgreSQL | Надёжность, поддержка JSON, транзакции, индексы, популярность |
| **Кэш** | Redis | Высокая производительность, поддержка структур данных |
| **Брокер событий** | Kafka | Высокая пропускная способность, долговременное хранение, replay событий |

### Фронтенд
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Фреймворк** | React + TypeScript | Популярность, экосистема, строгая типизация |
| **UI-библиотека** | Ant Design / Material-UI | Готовые компоненты, ускорение разработки |
| **Стейт-менеджмент** | Redux Toolkit / Zustand | Управление сложным состоянием |

### Инфраструктура и DevOps
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Контейнеризация** | Docker | Стандарт, изоляция, воспроизводимость |
| **Оркестрация** | Kubernetes | Масштабируемость, управление контейнерами, отказоустойчивость |
| **CI/CD** | GitHub Actions | Простота настройки, интеграция с репозиторием |
| **Мониторинг** | Prometheus + Grafana | Стандарт, гибкость, большое сообщество |
| **Логирование** | ELK Stack (или Loki) | Сбор, хранение, анализ логов |
| **Трассировка** | Jaeger / OpenTelemetry | Распределённая трассировка для микросервисов |

## 4. Архитектурные слои (высокоуровнево)

```
┌─────────────────────────────────────────────────────┐
│                    Presentation Layer               │
│           (HTTP REST API / gRPC / Web UI)           │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│                    Application Layer                 │
│              (Use Cases / Command & Queries)         │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│                    Domain Layer                      │
│            (Entities, Value Objects, Rules)          │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│                 Infrastructure Layer                 │
│   (Repositories, Adapters, Event Bus, External API)  │
└─────────────────────────────────────────────────────┘
```

- **Domain Layer** — ядро системы, независимо от инфраструктуры.
- **Application Layer** — оркестрация use cases, команды и запросы.
- **Presentation Layer** — взаимодействие с внешним миром (HTTP, gRPC, UI).
- **Infrastructure Layer** — реализации репозиториев, адаптеров, брокеров.

## 5. Компромиссы и обоснования

| Компромисс | Решение | Обоснование |
|------------|---------|-------------|
| **Монолит vs Микросервисы** | Начинаем с модульного монолита | Быстрый старт, простота транзакций, готовность к распилу |
| **Синхронность vs Асинхронность** | Команды — синхронно (gRPC), события — асинхронно (Kafka) | Команды требуют немедленного ответа; события — для слабой связанности |
| **CQRS vs CRUD** | Используем CQRS только для сложных read-моделей | Простота для MVP, расширяемость в будущем |
| **SQL vs NoSQL** | PostgreSQL как основная БД | Транзакции, целостность, гибкость JSON |
| **Глубокое тестирование vs Скорость** | Unit + Integration тесты обязательны, E2E — для ключевых сценариев | Баланс между качеством и скоростью |

## 6. Стратегия масштабирования

- **Вертикальное масштабирование** (увеличение ресурсов) — на ранних этапах.
- **Горизонтальное масштабирование** — после перехода на микросервисы.
- **Шардирование** данных — при превышении ёмкости одного инстанса БД.
- **Кэширование (Redis)** — для уменьшения нагрузки на БД.
- **Асинхронная обработка** — через Kafka для тяжёлых операций.

---

## 7. Стратегия безопасности

- **Аутентификация:** JWT (Access + Refresh токены).
- **Авторизация:** RBAC (роли: оператор, администратор, системный).
- **Шифрование:** TLS для всех внешних коммуникаций.
- **Секреты:** HashiCorp Vault (или Kubernetes Secrets).
- **Безопасность API:** Rate limiting, валидация входных данных.

---

## 8. Стратегия тестирования

| Уровень | Инструменты | Цель |
|---------|-------------|------|
| **Unit-тесты** | Go testing, testify | Проверка изолированных компонентов (Use Cases, Domain) |
| **Integration-тесты** | testcontainers, pgx | Проверка взаимодействия с БД, брокерами |
| **API-тесты** | Postman / Newman, gRPCurl | Валидация контрактов |
| **E2E-тесты** | Cypress (UI), Go (backend) | Критические пользовательские сценарии |
| **Performance-тесты** | Locust / K6 | Нагрузочное тестирование |

---

## 9. Стратегия мониторинга и алертинга

- **Метрики:** Время ответа, частота ошибок, использование ресурсов (CPU, память).
- **Логи:** Структурированные логи (JSON) с уровнем детализации.
- **Трассировка:** Распределённая трассировка для микросервисов.
- **Алерты:** В Slack / email при превышении порогов SLO.
